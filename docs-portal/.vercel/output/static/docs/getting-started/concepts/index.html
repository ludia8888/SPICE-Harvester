Core Concepts

SPICE Harvester is built around a strongly-typed ontology model inspired by Palantir Foundry. This page introduces the foundational concepts you need to understand before working with the platform.

Ontology​

The Ontology is the top-level container that defines the data model for your organization. It describes what types of objects exist, how they relate to each other, and what properties they carry.

An ontology in SPICE Harvester consists of:

  • Object Types -- blueprints for entities (e.g., Employee, Project, Asset)
  • Link Types -- relationships between object types (e.g., Employee manages Project)
  • Properties -- typed attributes on object types (e.g., fullName: string, startDate: date)

You can have multiple ontologies, each identified by a unique apiName. Most deployments use a single primary ontology.

Ontology: "acme-corp"
├── ObjectType: Employee
│   ├── Property: employeeId (string, primary key)
│   ├── Property: fullName (string)
│   ├── Property: department (string)
│   └── Property: startDate (date)
├── ObjectType: Project
│   ├── Property: projectId (string, primary key)
│   ├── Property: name (string)
│   └── Property: status (string)
└── LinkType: manages (Employee → Project)

Object Types​

An Object Type is a schema definition, similar to a table in a relational database or a class in object-oriented programming. Each object type declares:

  • apiName -- unique identifier used in API calls (e.g., Employee)
  • displayName -- human-readable label (e.g., "Employee")
  • primaryKey -- the property that uniquely identifies instances
  • properties -- a map of named, typed attributes

Object types are registered in the ontology and define the structure that all instances of that type must follow.

Link Types​

A Link Type defines a directed relationship between two object types. Links are first-class citizens in the ontology and enable graph-style traversal across your data.

Each link type specifies:

  • apiName -- unique identifier (e.g., manages)
  • sourceObjectType -- the "from" side of the relationship
  • targetObjectType -- the "to" side of the relationship
  • cardinality -- ONE_TO_ONE, ONE_TO_MANY, MANY_TO_ONE, or MANY_TO_MANY

Links are stored as edges in the data model and are queryable through the API. You can traverse links to navigate between related objects.

Properties​

A Property is a named, typed attribute on an object type. Properties define what data each instance carries.

Properties have:

  • apiName -- identifier used in API requests and responses
  • displayName -- human-readable label
  • dataType -- one of the supported data types above
  • required -- whether the property must be present on every instance
  • indexed -- whether the property is searchable in Elasticsearch

The primary key property is always required and indexed.

Instances​

An Instance is a concrete record of a particular object type. If an object type is the blueprint, an instance is the actual building. Each instance has:

  • __rid -- resource identifier, globally unique across the platform
  • __primaryKey -- the value of the primary key property
  • __apiName -- the object type this instance belongs to
  • properties -- the actual data values

Example instance of the Employee object type:

{
  "__rid": "ri.ontology.main.object.employee-00042",
  "__primaryKey": "EMP-042",
  "__apiName": "Employee",
  "properties": {
    "employeeId": "EMP-042",
    "fullName": "Jane Doe",
    "department": "Engineering",
    "startDate": "2023-06-15"
  }
}

Instances are stored in PostgreSQL (source of truth) and indexed in Elasticsearch (for search and analytics).

Actions​

Actions are the write side of the platform. Instead of direct CRUD mutations, SPICE Harvester uses an action system that provides:

  • Validation -- actions are validated against rules before execution
  • Audit trail -- every action is logged as an event in the event store
  • Undo support -- actions can be reversed through compensating actions
  • Conflict resolution -- concurrent modifications are detected and resolved
  • Templates -- reusable action definitions with dynamic expressions

Action Types​

Action TypeDescription
createObjectCreate a new instance of an object type
editObjectModify properties of an existing instance
deleteObjectRemove an instance
createLinkEstablish a relationship between two instances
deleteLinkRemove a relationship
bulkActionExecute multiple actions atomically

Action Templates​

Action templates use a dynamic expression language with operators like $ref, $now, $if, $switch, and $call to define reusable action logic. Templates are evaluated at execution time and can reference the current user, timestamp, and existing object properties.

Pipelines​

A Pipeline is a declarative data transformation workflow. Pipelines define how data flows from sources (files, APIs, databases) through a series of transforms into the ontology.

Pipelines support 22+ transform types including:

  • Structural: map, filter, flatten, groupBy, sort, distinct
  • Enrichment: lookup, join, geocode, resolve
  • Validation: validate, deduplicate, normalize
  • Output: upsertObjects, createLinks, publishEvent

Pipelines are defined as JSON or YAML configurations and executed by the Pipeline Runner workers.

Projections​

A Projection is a precomputed, denormalized view of data optimized for specific query patterns. Projections are built by the Projection Builder workers from the event stream.

Examples of projections:

  • An aggregated count of objects by type and department
  • A flattened view joining objects across link types
  • A time-series rollup of action activity

Projections are eventually consistent with the source of truth. The platform provides tools to monitor projection lag and trigger rebuilds.

Lineage​

Lineage tracks how data flows through the system. SPICE Harvester captures 15 edge types of lineage relationships:

Edge TypeDescription
DERIVED_FROMObject was derived from another object
IMPORTED_BYData was imported by a pipeline
TRANSFORMED_BYData was processed by a transform
LINKED_TOObject is linked to another object
CREATED_BYObject was created by an action
MODIFIED_BYObject was modified by an action
DELETED_BYObject was deleted by an action
VALIDATED_BYData was validated by a rule
ENRICHED_BYData was enriched by a lookup
MERGED_FROMObject was merged from multiple sources
SPLIT_INTOObject was split into multiple objects
AGGREGATED_FROMProjection was aggregated from source objects
PUBLISHED_TOData was published to an external system
CONSUMED_FROMData was consumed from an external system
SNAPSHOT_OFA point-in-time snapshot of an object

Lineage data is stored in the event store and can be queried through the API to trace data provenance.

Event Sourcing​

SPICE Harvester uses Event Sourcing as its persistence pattern. Every state change is captured as an immutable event:

  1. An action is submitted through the API
  2. The action is validated and converted to one or more events
  3. Events are published to Kafka for real-time processing
  4. Events are persisted to the S3/MinIO Event Store for durability
  5. Workers consume events to update read models (PostgreSQL, Elasticsearch, projections)

This design provides a complete audit trail, enables time-travel queries, and supports rebuilding any read model from the event history.
