Service Topology

SPICE Harvester consists of 33+ Docker services organized into four categories. This page catalogs every service with its port, dependencies, and purpose.

Core API Servers​

These are the externally accessible services that handle client requests.

ServicePortProtocolDescription
OMS (Ontology Management Service)8000HTTP/RESTPrimary API for ontology CRUD, object management, search, actions, and lineage
BFF (Backend-for-Frontend)8002HTTP/RESTAPI gateway handling authentication, routing, rate limiting, and response transformation
Funnel8003HTTP/RESTBulk data ingestion, file uploads, streaming imports, and pipeline management
Admin8004HTTP/RESTPlatform administration, user management, configuration, and health monitoring

Dependencies​

BFF  ──▶  OMS  ──▶  PostgreSQL, Elasticsearch, Redis, Kafka
     ──▶  Funnel ──▶  Kafka, MinIO
     ──▶  Admin  ──▶  PostgreSQL, Redis
     ──▶  Auth (internal)

Workers​

Workers are background services that consume events from Kafka and maintain the platform's read models and processing pipelines.

ServiceKafka Consumer GroupDescription
Event Processorspice-event-processorCore event handler; updates PostgreSQL and dispatches to downstream workers
Projection Builderspice-projection-builderBuilds and maintains denormalized projection views
Pipeline Runnerspice-pipeline-runnerExecutes data transformation pipelines
Lineage Workerspice-lineage-workerCaptures and stores data lineage edges
Schema Validatorspice-schema-validatorValidates incoming data against ontology schema definitions
Action Executorspice-action-executorProcesses deferred and scheduled actions
Notification Workerspice-notification-workerSends notifications (webhooks, email) on events
Retry Handlerspice-retry-handlerManages retry logic for failed event processing
Dead Letter Processorspice-dlpHandles events that have exhausted retry attempts
Metric Aggregatorspice-metric-aggregatorComputes and publishes platform metrics
Cache Warmerspice-cache-warmerPre-populates Redis caches for frequently accessed data
Index Syncerspice-index-syncerKeeps Elasticsearch indices synchronized with PostgreSQL
Snapshot Workerspice-snapshot-workerCreates periodic snapshots of event store state for faster replay

Worker Scaling​

Workers are horizontally scalable. Kafka consumer group rebalancing ensures that partitions are distributed evenly across worker instances:

# Scale the event processor to 3 replicas
docker compose up -d --scale event-processor=3
note

When scaling workers, ensure the number of Kafka partitions for each topic is at least equal to the number of worker replicas. The default partition count is 12.

Network Architecture​

All services communicate over a shared Docker network (spice-network). External access is only through the BFF gateway on port 8002.

Port Summary​

Port RangeCategory
8000-8004API Servers
5432, 9200, 6379, 9092, 2181, 9000Data Stores
8081, 8083Kafka Ecosystem
13000, 19090, 16686, 4317-4318, 9093, 3100Observability

Security Boundaries​

  • External: Only port 8002 (BFF) is exposed to the public network
  • Internal: All inter-service communication happens on the Docker bridge network
  • Data stores: Not exposed externally by default; access only from within the Docker network

Startup Order​

Services must start in a specific order due to dependencies. Docker Compose handles this through depends_on with health checks:

  1. Data stores: PostgreSQL, Elasticsearch, Redis, ZooKeeper, MinIO
  2. Kafka ecosystem: Kafka (depends on ZooKeeper), Schema Registry, Kafka Connect
  3. Core services: OMS (depends on PostgreSQL, Elasticsearch, Redis, Kafka), Funnel, Admin
  4. Gateway: BFF (depends on OMS, Funnel, Admin)
  5. Workers: All workers (depend on Kafka and relevant data stores)
  6. Observability: OTEL Collector, Prometheus, Jaeger, Loki, AlertManager, Grafana
