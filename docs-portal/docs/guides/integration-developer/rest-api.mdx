# REST API Integration

This guide walks integration developers through building applications that consume the Spice OS REST API. You will learn how to authenticate, query objects, execute actions, handle pagination, and implement robust error handling.

## API Architecture

Spice OS exposes **264 REST endpoints** (109 v2 + 155 v1) through the BFF (Backend for Frontend) service, organized into several API families:

| API Family | Prefix | Description |
|-----------|--------|-------------|
| **Foundry Ontology v2** | `/api/v2/ontologies` | Object types, objects, search, actions, link types |
| **Foundry Datasets v2** | `/api/v2/datasets` | Dataset CRUD, branches, transactions, files |
| **Foundry Connectivity v2** | `/api/v2/connectivity` | Connections, table imports, file imports |
| **Foundry Orchestration v2** | `/api/v2/orchestration` | Builds, schedules, jobs |
| **Internal BFF** | `/api/databases`, `/api/admin` | Database management, admin operations |
| **Lineage** | `/api/lineage` | Data lineage tracking |
| **Graph** | `/api/v1` | Graph traversal operations |

### Base URL

```
http://localhost:8080/api
```

In production, the BFF service is typically exposed behind a reverse proxy or API gateway.

## Authentication

All API requests (except health checks) require authentication via a Bearer token in the `Authorization` header:

```bash
curl -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  http://localhost:8080/api/v2/ontologies
```

Spice OS supports multiple authentication strategies:

| Strategy | Header | Use Case |
|----------|--------|----------|
| Bearer Token | `Authorization: Bearer <token>` | Standard API access |
| Agent Token | `Authorization: Bearer <agent-token>` + `X-Agent-Tool-ID` | Agent/tool integrations |
| User JWT | `Authorization: Bearer <jwt>` | SSO/OIDC authenticated users |
| Admin Token | `X-Admin-Token: <token>` | Administrative operations |

### OAuth Scope Enforcement

Certain v2 endpoints enforce **OAuth scopes** on the authenticated principal. If the principal's JWT does not include the required scope, the API returns a Foundry-style error:

```json
{
  "errorCode": "PERMISSION_DENIED",
  "errorName": "ApiUsageDenied",
  "errorInstanceId": "...",
  "parameters": { "missingScope": "api:datasets-write" }
}
```

See the [Auth Flow Guide](./auth-flow) for the full list of platform API scopes and authentication patterns.

## Object Operations

### List Object Types

```bash
GET /api/v2/ontologies/{ontologyApiName}/objectTypes
```

**Response:**

```json
{
  "data": [
    {
      "apiName": "Employee",
      "displayName": "Employee",
      "description": "A person employed by the organization",
      "primaryKey": "employeeId",
      "properties": {
        "employeeId": { "dataType": { "type": "string" } },
        "fullName": { "dataType": { "type": "string" } },
        "department": { "dataType": { "type": "string" } },
        "startDate": { "dataType": { "type": "date" } }
      },
      "rid": "ri.spice.main.object-type.Employee"
    }
  ]
}
```

```bash
GET /api/v2/ontologies/{ontologyApiName}/objectTypes/{objectTypeApiName}/objects/{primaryKey}
```

### Create Object (via Actions)

Objects are created through the Actions API, not direct POST:

```bash
POST /api/v2/ontologies/{ontologyApiName}/actions/createObject
Content-Type: application/json

{
  "parameters": {
    "objectType": "Employee",
    "properties": {
      "employeeId": "EMP-099",
      "fullName": "John Smith",
      "department": "Marketing",
      "startDate": "2024-01-15"
    }
  }
}
```

### Edit Object

## Search API

The Search API uses the `SearchJsonQueryV2` query language to filter, sort, and paginate objects.

### Basic Search

```bash
POST /api/v2/ontologies/{ontologyApiName}/objectTypes/{objectTypeApiName}/objects/search
Content-Type: application/json

{
  "where": {
    "type": "eq",
    "field": "department",
    "value": "Engineering"
  },
  "pageSize": 25,
  "orderBy": {
    "fields": [
      { "field": "fullName", "direction": "asc" }
    ]
  }
}
```

### Search Operators

Spice OS supports **23 search operators**:

#### Comparison Operators

| Operator | Description | Example |
|----------|-------------|---------|
| `eq` | Equal to | `{"type": "eq", "field": "status", "value": "ACTIVE"}` |
| `gt` | Greater than | `{"type": "gt", "field": "salary", "value": 50000}` |
| `gte` | Greater than or equal | `{"type": "gte", "field": "age", "value": 18}` |
| `lt` | Less than | `{"type": "lt", "field": "age", "value": 65}` |
| `lte` | Less than or equal | `{"type": "lte", "field": "score", "value": 100}` |

#### Null Check

| Operator | Description | Example |
|----------|-------------|---------|
| `isNull` | Check if null | `{"type": "isNull", "field": "email", "value": true}` |

#### Collection Operators

| Operator | Description | Example |
|----------|-------------|---------|
| `contains` | Array contains value | `{"type": "contains", "field": "tags", "value": "urgent"}` |
| `in` | Value in set | `{"type": "in", "field": "status", "value": ["ACTIVE", "PENDING"]}` |

#### Text Search Operators

| Operator | Description | Example |
|----------|-------------|---------|
| `containsAnyTerm` | Contains any term | `{"type": "containsAnyTerm", "field": "name", "value": "jane john"}` |
| `containsAllTerms` | Contains all terms | `{"type": "containsAllTerms", "field": "bio", "value": "engineer senior"}` |
| `containsAllTermsInOrder` | Terms in exact order | `{"type": "containsAllTermsInOrder", "field": "title", "value": "senior engineer"}` |
| `containsAllTermsInOrderPrefixLastTerm` | Prefix match on last term | Ideal for typeahead/autocomplete |
| `startsWith` | Starts with prefix | `{"type": "startsWith", "field": "name", "value": "Jan"}` |
| `wildcard` | Wildcard pattern | `{"type": "wildcard", "field": "email", "value": "*@acme.com"}` |
| `regex` | Regex pattern | `{"type": "regex", "field": "code", "value": "^EMP-\\d{3}$"}` |

#### Date Range

| Operator | Description |
|----------|-------------|
| `relativeDateRange` | Relative date range with ISO 8601 durations |

#### Geospatial (Point)

| Operator | Description |
|----------|-------------|
| `withinDistanceOf` | Point within radius |
| `withinBoundingBox` | Point within rectangle |
| `withinPolygon` | Point within polygon |

#### Geospatial (Shape)

| Operator | Description |
|----------|-------------|
| `intersectsBoundingBox` | Shape overlaps rectangle |
| `intersectsPolygon` | Shape overlaps polygon |
| `doesNotIntersectBoundingBox` | Shape outside rectangle |
| `doesNotIntersectPolygon` | Shape outside polygon |

#### Logical Operators

Combine queries with `and`, `or`, and `not`:

```json
{
  "type": "and",
  "value": [
    { "type": "eq", "field": "department", "value": "Engineering" },
    { "type": "gte", "field": "salary", "value": 80000 },
    {
      "type": "or",
      "value": [
        { "type": "eq", "field": "level", "value": "Senior" },
        { "type": "eq", "field": "level", "value": "Staff" }
      ]
    }
  ]
}
```

:::caution
Query nesting is limited to **3 levels** maximum depth. Deeply nested queries will be rejected.
:::

### Selective Fields

Request only the properties you need with `select`:

```json
{
  "where": { "type": "eq", "field": "department", "value": "Engineering" },
  "select": ["employeeId", "fullName", "salary"],
  "pageSize": 50
}
```

## Pagination

Spice OS uses **cursor-based pagination** with opaque page tokens.

### Paginating Through Results

```python
import requests

base_url = "http://localhost:8080/api/v2/ontologies/acme/objectTypes/Employee/objects"
headers = {"Authorization": "Bearer YOUR_TOKEN"}
params = {"pageSize": 25}

all_objects = []
page_token = None

while True:
    if page_token:
        params["pageToken"] = page_token

    response = requests.get(base_url, headers=headers, params=params)
    data = response.json()

    all_objects.extend(data["data"])

    page_token = data.get("nextPageToken")
    if not page_token:
        break

print(f"Total objects: {len(all_objects)}")
```

### Pagination Best Practices

- **Use consistent page sizes** -- don't change `pageSize` between pages
- **Don't modify data while paginating** -- cursor stability is not guaranteed across mutations
- **Page tokens are opaque** -- don't parse or construct them manually
- **Tokens expire** -- re-start pagination if a token becomes invalid


```bash
# Search linked objects

  "sourceObjectRid": "ri.spice.main.object.employee-00042",
  "where": { "type": "eq", "field": "status", "value": "ACTIVE" },
  "pageSize": 25
}
```

## Error Handling

### Error Response Format

All errors follow a structured format:

```json
{
  "errorCode": "ObjectNotFound",
  "errorName": "ObjectNotFound",
  "errorInstanceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "parameters": {
    "objectType": "Employee",
    "primaryKey": "EMP-999"
  }
}
```

:::note
All `/api/v2/` endpoints return errors using the Foundry-compatible `FoundryAPIError` envelope shown above. This is handled by a global exception handler registered on the BFF service. The `errorInstanceId` is a UUID generated per error occurrence for tracing.
:::

### Common Error Codes

| HTTP Status | Error Code | Description |
|------------|------------|-------------|
| 400 | `InvalidRequest` | Malformed request body or parameters |
| 401 | `Unauthorized` | Missing or invalid authentication |
| 403 | `PermissionDenied` | Insufficient permissions |
| 404 | `ObjectNotFound` | Object does not exist |
| 404 | `ObjectTypeNotFound` | Object type does not exist |
| 409 | `ConflictError` | Concurrent modification conflict |
| 429 | `RateLimitExceeded` | Too many requests |
| 500 | `InternalError` | Server-side error |

### Retry Strategy

Implement exponential backoff for transient errors:

```python
import time
import requests

def api_request_with_retry(url, headers, max_retries=3, base_delay=1):
    for attempt in range(max_retries + 1):
        response = requests.get(url, headers=headers)

        if response.status_code == 429:
            retry_after = int(response.headers.get("Retry-After", base_delay * (2 ** attempt)))
            time.sleep(retry_after)
            continue

        if response.status_code >= 500:
            if attempt < max_retries:
                time.sleep(base_delay * (2 ** attempt))
                continue

        return response

    return response
```

**Retry these status codes:** `429`, `500`, `502`, `503`, `504`

**Do NOT retry:** `400`, `401`, `403`, `404`, `409`

## Health Check

Monitor the API availability:

```bash
# Service metadata
GET /api/

# Health check with dependency status
GET /api/health
```

**Response:**

```json
{
  "data": {
    "oms_connected": true,
    "status": "healthy"
  }
}
```

| Status | Meaning |
|--------|---------|
| `healthy` | All dependencies available |
| `degraded` | OMS unavailable but BFF still serving cached/partial data |

## Performance Tips

1. **Use `select`** -- request only needed properties to reduce payload size
2. **Set appropriate `pageSize`** -- balance between request count and response size (25-100 typical)
3. **Cache object type metadata** -- object type definitions change infrequently
4. **Use search instead of list + filter** -- let the server filter with Elasticsearch
5. **Batch writes** -- use batch action endpoints for multiple mutations
6. **Connection pooling** -- reuse HTTP connections for sequential requests
