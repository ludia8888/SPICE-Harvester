# Core Concepts

Spice OS is built around a strongly-typed ontology model inspired by Palantir Foundry. This page introduces the foundational concepts you need to understand before working with the platform.

## Ontology

The **Ontology** is the top-level container that defines the data model for your organization. It describes what types of objects exist, how they relate to each other, and what properties they carry.

An ontology in Spice OS consists of:
- **Object Types** -- blueprints for entities (e.g., Employee, Project, Asset)
- **Link Types** -- relationships between object types (e.g., Employee *manages* Project)
- **Properties** -- typed attributes on object types (e.g., `fullName: string`, `startDate: date`)

You can have multiple ontologies, each identified by a unique `apiName`. Most deployments use a single primary ontology.

```
Ontology: "acme-corp"
├── ObjectType: Employee
│   ├── Property: employeeId (string, primary key)
│   ├── Property: fullName (string)
│   ├── Property: department (string)
│   └── Property: startDate (date)
├── ObjectType: Project
│   ├── Property: projectId (string, primary key)
│   ├── Property: name (string)
│   └── Property: status (string)
└── LinkType: manages (Employee → Project)
```

## Object Types

An **Object Type** is a schema definition, similar to a table in a relational database or a class in object-oriented programming. Each object type declares:

- **apiName** -- unique identifier used in API calls (e.g., `Employee`)
- **displayName** -- human-readable label (e.g., "Employee")
- **primaryKey** -- the property that uniquely identifies instances
- **properties** -- a map of named, typed attributes

Object types are registered in the ontology and define the structure that all instances of that type must follow.

### Supported Data Types

| Type | Description | Example |
|------|-------------|---------|
| `string` | UTF-8 text | `"Jane Doe"` |
| `integer` | 64-bit signed integer | `42` |
| `long` | 64-bit signed integer (alias) | `9876543210` |
| `double` | 64-bit floating point | `3.14159` |
| `boolean` | True or false | `true` |
| `date` | ISO 8601 date | `"2024-01-15"` |
| `timestamp` | ISO 8601 datetime | `"2024-01-15T09:30:00Z"` |
| `geopoint` | Latitude/longitude pair | `{"lat": 37.7749, "lon": -122.4194}` |
| `geoshape` | GeoJSON geometry | `{"type": "Polygon", ...}` |
| `array` | Ordered list of a base type | `["tag1", "tag2"]` |

## Link Types

A **Link Type** defines a directed relationship between two object types. Links are first-class citizens in the ontology and enable graph-style traversal across your data.

Each link type specifies:
- **apiName** -- unique identifier (e.g., `manages`)
- **sourceObjectType** -- the "from" side of the relationship
- **targetObjectType** -- the "to" side of the relationship
- **cardinality** -- `ONE_TO_ONE`, `ONE_TO_MANY`, `MANY_TO_ONE`, or `MANY_TO_MANY`

Links are stored as edges in the data model and are queryable through the API. You can traverse links to navigate between related objects.

## Properties

A **Property** is a named, typed attribute on an object type. Properties define what data each instance carries.

Properties have:
- **apiName** -- identifier used in API requests and responses
- **displayName** -- human-readable label
- **dataType** -- one of the supported data types above
- **required** -- whether the property must be present on every instance
- **indexed** -- whether the property is searchable in Elasticsearch

The primary key property is always required and indexed.

## Instances

An **Instance** is a concrete record of a particular object type. If an object type is the blueprint, an instance is the actual building. Each instance has:

- **\_\_rid** -- resource identifier, globally unique across the platform
- **\_\_primaryKey** -- the value of the primary key property
- **\_\_apiName** -- the object type this instance belongs to
- **properties** -- the actual data values

Example instance of the `Employee` object type:

```json
{
  "__rid": "ri.ontology.main.object.employee-00042",
  "__primaryKey": "EMP-042",
  "__apiName": "Employee",
  "properties": {
    "employeeId": "EMP-042",
    "fullName": "Jane Doe",
    "department": "Engineering",
    "startDate": "2023-06-15"
  }
}
```

Instances are stored in PostgreSQL (source of truth) and indexed in Elasticsearch (for search and analytics).

## Actions

**Actions** are the write side of the platform. Instead of direct CRUD mutations, Spice OS uses an action system that provides:

- **Validation** -- actions are validated against rules before execution
- **Audit trail** -- every action is logged as an event in the event store
- **Undo support** -- actions can be reversed through compensating actions
- **Conflict resolution** -- concurrent modifications are detected and resolved
- **Templates** -- reusable action definitions with dynamic expressions

### Action Types

| Action Type | Description |
|-------------|-------------|
| `createObject` | Create a new instance of an object type |
| `editObject` | Modify properties of an existing instance |
| `deleteObject` | Remove an instance |
| `createLink` | Establish a relationship between two instances |
| `deleteLink` | Remove a relationship |
| `bulkAction` | Execute multiple actions atomically |

### Action Templates

Action templates use a dynamic expression language with operators like `$ref`, `$now`, `$if`, `$switch`, and `$call` to define reusable action logic. Templates are evaluated at execution time and can reference the current user, timestamp, and existing object properties.

## Datasets

A **Dataset** is a versioned, immutable data object stored on LakeFS (backed by MinIO/S3). Datasets power the ontology by providing the raw data that gets objectified into instances.

Key dataset operations:
- **Branches** -- Git-like branching for parallel data development
- **Transactions** -- Atomic read-modify-write operations with OPEN/COMMITTED/ABORTED lifecycle
- **Versioning** -- Every change creates a new version with a LakeFS commit ID
- **File storage** -- Datasets contain files (CSV, Parquet, etc.) addressable by path

```
Dataset: "sales-orders"
├── Branch: main
│   ├── Version: v3 (commit: abc123)
│   │   └── data.csv (180,000 rows)
│   ├── Version: v2 (commit: def456)
│   └── Version: v1 (commit: ghi789)
└── Branch: staging
    └── Version: v1 (commit: jkl012)
```

Datasets are managed through the **Foundry Datasets v2 API** with 19 endpoints covering CRUD, branching, transactions, file management, and schema operations.

## Connections & Connectors

A **Connection** represents a configured link to an external data source. Spice OS supports 6 connector types:

| Connector | Library | Use Case |
|-----------|---------|----------|
| **Google Sheets** | gspread | Spreadsheet imports |
| **PostgreSQL** | asyncpg | Relational databases |
| **MySQL** | pymysql | Relational databases |
| **Oracle** | oracledb | Enterprise databases |
| **Snowflake** | snowflake-connector | Data warehouses |
| **SQL Server** | pymssql | Enterprise databases |

Each connection stores:
- **Public configuration** (host, port, database) -- in `connector_sources.config_json`
- **Secrets** (passwords, connection strings) -- **AES-GCM encrypted** in `connector_connection_secrets`

Connections support three resource types:
- **Table Imports** -- SQL query-based data ingestion
- **File Imports** -- File-based data ingestion
- **Virtual Tables** -- Read-only query views (SNAPSHOT only)

### Sync Modes

When importing data, you choose a sync mode that determines how new data merges with existing data:

| Mode | Strategy | When to Use |
|------|----------|-------------|
| **SNAPSHOT** | Replace all data each run | Daily reports, small tables |
| **APPEND** | Add new rows (deduplicate by hash) | Event logs, growing datasets |
| **UPDATE** | Upsert by primary key | Customer profiles, status tables |
| **INCREMENTAL** | Fetch only rows after watermark | Large tables with timestamps |
| **CDC** | Track changes via database tokens (SCN, binlog) | Real-time replication |

## Pipelines

A **Pipeline** is a declarative data transformation workflow defined as a DAG (directed acyclic graph) of nodes. Pipelines define how data flows from sources through transforms into outputs.

### Pipeline Nodes

```
[Input Nodes] → [Transform Nodes] → [Output Nodes]
```

- **Input** -- data sources (datasets, JDBC, Kafka, S3)
- **Transform** -- 38 operations (join, filter, compute, aggregate, window, UDF, etc.)
- **Output** -- result sinks (dataset, ontology, geotemporal)

### Execution Modes

| Mode | Purpose | Side Effects |
|------|---------|-------------|
| **Preview** | In-memory simulation with sample data | None |
| **Build** | Full execution producing artifacts | Creates artifacts in S3 |
| **Deploy** | Promotes artifacts to production | Writes to output dataset + ontology |

### User-Defined Functions (UDFs)

UDFs allow custom per-row transformations in a **sandboxed** environment. Only safe Python builtins are allowed -- no imports, loops, or system access.

```python
def transform(row):
    return row.get("name", "").upper()
```

### Proposals

Changes to pipelines on protected branches (`main`, `prod`) require **proposal approval** before deployment. Proposals capture an immutable snapshot of the build output, schema, and ontology state.

## Orchestration & Scheduling

**Orchestration** manages pipeline execution lifecycle through builds, schedules, and runs.

- **Builds** -- create, monitor, cancel pipeline execution jobs
- **Schedules** -- define cron expressions or intervals for recurring execution
- **Runs** -- track execution history with status and output

The **Pipeline Scheduler** is an independent background process that polls the registry every N seconds, evaluates cron expressions, checks pipeline dependencies, and enqueues jobs to Kafka.

Schedules can be **paused** and **unpaused** without deleting the configuration.

## Security

Spice OS provides multiple security layers:

| Layer | Protection |
|-------|-----------|
| **Secret Encryption** | AES-GCM with AAD (Additional Authenticated Data) per source |
| **SQL Injection Guard** | All JDBC queries normalized; semicolons and multi-statements rejected |
| **Query Timeouts** | 300-second timeout on all JDBC adapter operations |
| **Feature Flags** | JDBC and CDC features gated per database |
| **Plaintext Block** | Production environments **cannot** store secrets without encryption keys |
| **UDF Sandbox** | AST validation blocks imports, loops, and system access |

## Projections

A **Projection** is a precomputed, denormalized view of data optimized for specific query patterns. Projections are built by the Projection Builder workers from the event stream.

Examples of projections:
- An aggregated count of objects by type and department
- A flattened view joining objects across link types
- A time-series rollup of action activity

Projections are eventually consistent with the source of truth. The platform provides tools to monitor projection lag and trigger rebuilds.

## Lineage

**Lineage** tracks how data flows through the system. Spice OS captures 15 edge types of lineage relationships:

| Edge Type | Description |
|-----------|-------------|
| `DERIVED_FROM` | Object was derived from another object |
| `IMPORTED_BY` | Data was imported by a pipeline |
| `TRANSFORMED_BY` | Data was processed by a transform |
| `LINKED_TO` | Object is linked to another object |
| `CREATED_BY` | Object was created by an action |
| `MODIFIED_BY` | Object was modified by an action |
| `DELETED_BY` | Object was deleted by an action |
| `VALIDATED_BY` | Data was validated by a rule |
| `ENRICHED_BY` | Data was enriched by a lookup |
| `MERGED_FROM` | Object was merged from multiple sources |
| `SPLIT_INTO` | Object was split into multiple objects |
| `AGGREGATED_FROM` | Projection was aggregated from source objects |
| `PUBLISHED_TO` | Data was published to an external system |
| `CONSUMED_FROM` | Data was consumed from an external system |
| `SNAPSHOT_OF` | A point-in-time snapshot of an object |

Lineage data is stored in the event store and can be queried through the API to trace data provenance.

## Event Sourcing

Spice OS uses **Event Sourcing** as its persistence pattern. Every state change is captured as an immutable event:

1. An action is submitted through the API
2. The action is validated and converted to one or more events
3. Events are published to **Kafka** for real-time processing
4. Events are persisted to the **S3/MinIO Event Store** for durability
5. Workers consume events to update **read models** (PostgreSQL, Elasticsearch, projections)

This design provides a complete audit trail, enables time-travel queries, and supports rebuilding any read model from the event history.

## How Concepts Relate

```
┌─────────────────────────────────────────────────────────────┐
│                         Ontology                            │
│  ┌──────────────┐    ┌──────────────┐                       │
│  │  ObjectType   │───▶│   LinkType   │                       │
│  │  (blueprint)  │    │ (relationship)│                      │
│  └──────┬───────┘    └──────────────┘                       │
│         │                                                   │
│  ┌──────▼───────┐                                           │
│  │  Properties   │                                           │
│  │  (attributes) │                                           │
│  └──────────────┘                                           │
└─────────────────────────────────────────────────────────────┘
         │ defines structure for
         ▼
┌─────────────────┐     ┌─────────────┐     ┌──────────────┐
│    Instances     │────▶│   Actions   │     │  Connections  │
│  (actual data)   │◀────│  (mutations) │     │ (ext. sources)│
└────────┬────────┘     └──────┬──────┘     └──────┬───────┘
         │                     │ emit               │ import
         │              ┌──────▼──────┐      ┌──────▼───────┐
         │              │   Events    │      │   Datasets   │
         │              └──────┬──────┘      │  (versioned) │
         │                     │ consumed    └──────┬───────┘
         │              ┌──────▼──────┐             │
         └─────────────▶│ Projections │◀────────────┘
                        │  Pipelines  │
                        │  Lineage    │
                        └─────────────┘
```
