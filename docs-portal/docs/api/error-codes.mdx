# Error Codes Reference

SPICE Harvester defines 290+ structured error codes across all subsystems. This page documents the error format, naming conventions, severity levels, retry policies, and common error codes.

## Error Response Format

All API errors return a structured JSON response:

```json
{
  "errorCode": "NOT_FOUND",
  "errorName": "ObjectNotFound",
  "errorInstanceId": "err-2024-01-15-a1b2c3d4",
  "parameters": {
    "objectType": "Employee",
    "primaryKey": "EMP-999"
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `errorCode` | string | Machine-readable code following the taxonomy structure |
| `errorName` | string | Human-readable error name in PascalCase |
| `errorInstanceId` | string | Unique identifier for this error occurrence. Use this when contacting support. |
| `parameters` | object | Context-specific details for debugging. Contents vary by error type. |

## Error Code Structure

Error codes follow the pattern:

```
SHV-{SUBSYSTEM}-{DOMAIN}-{CLASS}-{NUMBER}
```

| Segment | Description | Examples |
|---------|-------------|---------|
| `SHV` | Platform prefix (SPICE Harvester) | Always `SHV` |
| `SUBSYSTEM` | Service subsystem | `ONT`, `ACT`, `SRH`, `PIP`, `LIN`, `AUTH`, `SYS` |
| `DOMAIN` | Functional domain | `OBJ`, `SCH`, `QRY`, `IDX`, `EVT`, `CFG` |
| `CLASS` | Error classification | `VAL`, `NFD`, `CON`, `PRM`, `INT`, `TMO` |
| `NUMBER` | Sequential number | `001`-`999` |

**Example:** `SHV-ONT-OBJ-NFD-001` = SPICE Harvester, Ontology subsystem, Object domain, Not Found class, error #001.

## Error Classes

| Code | Class | HTTP Status | Retryable | Description |
|------|-------|-------------|-----------|-------------|
| `VAL` | Validation | 400 / 422 | No | Input validation failure |
| `NFD` | Not Found | 404 | No | Resource does not exist |
| `CON` | Conflict | 409 | Yes (with re-read) | Concurrent modification detected |
| `PRM` | Permission | 403 | No | Insufficient permissions |
| `INT` | Internal | 500 | Yes | Unexpected internal error |
| `TMO` | Timeout | 504 | Yes | Operation timed out |
| `LIM` | Rate Limit | 429 | Yes (after delay) | Rate limit exceeded |
| `UNV` | Unavailable | 503 | Yes | Service temporarily unavailable |
| `DUP` | Duplicate | 409 | No | Duplicate resource |
| `DEP` | Dependency | 424 | Yes | Downstream dependency failure |

## Severity Levels

Each error carries a severity level that determines logging and alerting behavior:

| Level | Description | Alerting |
|-------|-------------|----------|
| `INFO` | Expected error from normal usage (e.g., object not found) | None |
| `WARNING` | Unusual condition that may indicate a problem (e.g., approaching rate limit) | Log warning |
| `ERROR` | Failure that prevents the request from completing | Alert on sustained rate |
| `CRITICAL` | System-level failure requiring immediate attention | Immediate page |

## Retry Policies

### General Retry Strategy

For retryable errors, use exponential backoff with jitter:

```
delay = min(base_delay * 2^attempt + random_jitter, max_delay)
```

| Parameter | Default | Description |
|-----------|---------|-------------|
| `base_delay` | 100 ms | Initial retry delay |
| `max_delay` | 30 s | Maximum delay between retries |
| `max_attempts` | 3 | Maximum number of retry attempts |
| `jitter` | 0-100 ms | Random jitter to prevent thundering herd |

### Per-Class Retry Policies

| Error Class | Retryable | Strategy | Notes |
|-------------|-----------|----------|-------|
| `VAL` | No | -- | Fix the request payload |
| `NFD` | No | -- | Verify the resource identifier |
| `CON` | Yes | Re-read + retry | Read the latest version, then resubmit |
| `PRM` | No | -- | Check user permissions |
| `INT` | Yes | Exponential backoff | Wait and retry |
| `TMO` | Yes | Exponential backoff | Consider reducing request scope |
| `LIM` | Yes | Respect `Retry-After` header | Wait for the specified duration |
| `UNV` | Yes | Exponential backoff | Service is temporarily down |
| `DUP` | No | -- | Resource already exists |
| `DEP` | Yes | Exponential backoff | Downstream service issue |

## Handling Errors in Client Code

### Python Example

```python
import requests
import time

def call_api_with_retry(url, headers, max_retries=3):
    for attempt in range(max_retries):
        response = requests.get(url, headers=headers)

        if response.status_code == 200:
            return response.json()

        error = response.json()
        error_class = error["errorCode"].split("-")[4] if "-" in error["errorCode"] else ""

        # Non-retryable errors
        if error_class in ("VAL", "NFD", "PRM", "DUP"):
            raise ApiError(error)

        # Rate limited - respect Retry-After header
        if response.status_code == 429:
            wait = int(response.headers.get("Retry-After", 30))
            time.sleep(wait)
            continue

        # Retryable errors - exponential backoff
        delay = min(0.1 * (2 ** attempt), 30)
        time.sleep(delay)

    raise ApiError({"errorCode": "MAX_RETRIES", "errorName": "MaxRetriesExceeded"})
```
