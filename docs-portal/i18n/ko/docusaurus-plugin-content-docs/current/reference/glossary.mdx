# 용어집

Spice OS 플랫폼 전반에서 사용되는 주요 용어와 개념을 정리한 레퍼런스입니다.

---

**Action**
온톨로지를 변경하는 쓰기 작업입니다. 객체/링크 생성, 수정, 삭제가 액션에 포함됩니다. 모든 액션은 실행 전에 검증되며, 이벤트로 기록되고, undo를 지원합니다. [Actions API](/docs/api/v2-actions)를 참고하세요.

**Action Template**
재사용 가능한 파라미터 기반 액션 정의입니다. `$ref`, `$now`, `$if`, `$switch`, `$call` 같은 동적 표현식을 사용해 실행 시점에 액션을 생성합니다. 템플릿은 현재 사용자와 대상 객체 컨텍스트에서 평가됩니다.

**Aggregate**
이벤트 소싱에서 aggregate는 정합성 관점에서 하나의 단위로 취급되는 도메인 객체들의 묶음입니다. Spice OS에서는 각 객체 인스턴스가 aggregate이며 RID로 식별됩니다.

**BFF (Backend-for-Frontend)**
외부 클라이언트 트래픽을 처리하는 API 게이트웨이 서비스입니다. BFF는 8002 포트에서 실행되며 인증/라우팅/레이트 리미팅/응답 변환을 수행합니다. 외부 네트워크에 노출되는 유일한 서비스입니다.

**CQRS (Command Query Responsibility Segregation)**
읽기와 쓰기 작업을 서로 다른 경로로 분리하는 아키텍처 패턴입니다. Spice OS는 CQRS를 사용해 읽기 측(Elasticsearch, Redis)과 쓰기 측(PostgreSQL, Kafka)을 독립적으로 스케일/최적화할 수 있습니다.

**Data Type**
속성 값의 타입입니다. 지원 타입에는 `string`, `integer`, `long`, `double`, `boolean`, `date`, `timestamp`, `geopoint`, `geoshape`, `array` 등이 있습니다.

**Dead Letter Queue (DLQ)**
모든 재시도를 소진한 이벤트가 전송되는 Kafka 토픽(`spice.events.dlq`)입니다. DLQ의 이벤트는 수동 점검과 해결이 필요합니다.

**Event**
상태 변경의 불변(immutable) 기록입니다. 이벤트는 이벤트 소싱 아키텍처에서 영속성의 원자 단위입니다. 각 이벤트는 타입, aggregate ID, 시퀀스 번호, 타임스탬프, payload, metadata를 가집니다.

**Event Sourcing**
가변 스냅샷으로 상태를 저장하는 대신, 불변 이벤트 시퀀스에서 상태를 파생하는 영속성 패턴입니다. Spice OS는 모든 이벤트를 S3/MinIO 이벤트 스토어에 저장하고, 리드 모델을 이벤트에서 파생합니다.

**Event Store**
S3/MinIO에 저장되는 내구성 있는 append-only 이벤트 로그입니다. 이벤트 스토어는 정합성의 기준(canonical source of truth)이며, 다른 데이터 스토어는 모두 이벤트 스토어의 프로젝션입니다.

**FoundryAPIError**
모든 `/api/v2/` 엔드포인트에서 사용하는 구조화된 예외 클래스 및 에러 엔벨로프 형식입니다. `errorCode`, `errorName`, `errorInstanceId`(UUID), `parameters`를 포함하는 JSON을 반환합니다. BFF 서비스에 등록된 전역 `foundry_exception_handler`가 처리합니다. [API 개요](/docs/api/overview)를 참고하세요.

**Funnel**
BFF 데이터 커넥터 플로우에서 사용하는 내부 테이블형 타입 추론/시트 구조 분석 런타임입니다. Foundry 정렬 토폴로지에서는 standalone 공개 서비스로 노출되지 않습니다.

**Instance**
특정 객체 유형의 실제 레코드(행)입니다. 각 인스턴스는 전역적으로 유일한 RID, 기본키, 그리고 객체 유형 스키마에 맞는 속성 값 집합을 가집니다.

**Kafka**
서비스 간 메시징에 사용되는 분산 이벤트 스트리밍 플랫폼입니다. Kafka는 이벤트 분산의 백본(backbone) 역할을 하며, 모든 컨슈머 워커에 at-least-once 전달을 보장합니다.

**Lineage**
데이터가 시스템을 통해 어떻게 흐르는지에 대한 기록입니다. 라인리지는 `DERIVED_FROM`, `IMPORTED_BY`, `TRANSFORMED_BY` 등 15개 엣지 타입을 추적하여 전체 데이터 출처(provenance)를 제공합니다. [핵심 개념](/docs/getting-started/concepts)을 참고하세요.

**Link Type**
두 객체 유형 사이의 방향성 있는 관계 정의입니다. 링크 유형은 source/target 객체 유형과 cardinality(`ONE_TO_ONE`, `ONE_TO_MANY`, `MANY_TO_ONE`, `MANY_TO_MANY`)를 지정합니다.

**MinIO**
이벤트 스토어, 파일 저장, 파이프라인 아티팩트에 사용되는 S3 호환 오브젝트 스토리지입니다. 9000 포트(API)와 9001 포트(console)에서 실행됩니다.

**Object Type**
온톨로지에서 엔티티 범주에 대한 스키마 정의입니다. 객체 유형은 속성(이름/타입/required/indexed)과 인스턴스가 따라야 할 기본키를 정의합니다. 관계형 DB의 테이블과 유사합니다.

**OMS (Ontology Management Service)**
온톨로지 관리, 객체 CRUD, 검색, 액션, 라인리지의 핵심 백엔드 서비스입니다. 8000 포트에서 실행됩니다. OMS는 비즈니스 로직을 구현하는 중심 서비스입니다.

**Ontology**
데이터 모델을 정의하는 최상위 컨테이너입니다. 온톨로지는 객체 유형과 링크 유형, 그리고 그 관계를 포함합니다. 각 온톨로지는 고유한 API 이름으로 식별됩니다.

**OAuth Scope**
Spice OS에서 API 작업을 제한하기 위해 사용하는 접근 제어 메커니즘입니다. scope는 JWT의 `scope` 또는 `scp` 클레임에서 추출됩니다. SPICE는 8개의 플랫폼 API scope(예: `api:datasets-read`, `api:ontologies-write`)를 정의합니다. [RBAC 가이드](/docs/guides/platform-admin/rbac)를 참고하세요.

**Outbox Pattern**
Kafka에 이벤트를 신뢰성 있게 게시하기 위한 기법입니다. 변이와 동일한 DB 트랜잭션 안에서 이벤트를 outbox 테이블에 기록하고, relay 프로세스가 커밋된 이벤트를 읽어 Kafka로 게시합니다.

**Page Token**
v2 API에서 사용하는 커서 기반 페이지네이션용 base64 인코딩된 opaque 문자열입니다. `nextPageToken` 필드로 반환되며, 다음 페이지를 가져오기 위해 다시 전달해야 합니다.

**Pipeline**
소스에서 읽고 변환을 적용한 뒤 결과를 온톨로지에 쓰는 선언적 데이터 변환 워크플로입니다. 파이프라인은 22개 이상의 변환 타입을 지원하며 Pipeline Runner 워커가 실행합니다.

**Platform API Scope**
주요 API 패밀리에 대한 접근을 제어하는 8개의 사전 정의 scope입니다: `api:datasets-read`, `api:datasets-write`, `api:ontologies-read`, `api:ontologies-write`, `api:orchestration-read`, `api:orchestration-write`, `api:connectivity-read`, `api:connectivity-write`. 보호된 엔드포인트에서 `require_scopes` 의존성으로 강제됩니다.

**Primary Key**
객체 유형 내에서 인스턴스를 고유하게 식별하는 속성입니다. 기본키 값은 API URL에 사용되며 v2 응답에서 `__primaryKey` 로 반환됩니다.

**Projection**
특정 쿼리 패턴에 최적화된 사전 계산된 비정규화 뷰입니다. 프로젝션은 Projection Builder 워커가 이벤트 스트림에서 빌드하며 최종 일관성을 가집니다.

**Property**
객체 유형의 이름/타입이 지정된 속성입니다. 각 속성은 API 이름, display name, data type, required/indexed 플래그를 가집니다.

**Read Model**
쿼리 성능에 최적화된 데이터 스토어입니다. CQRS에서 리드 모델은 쓰기 측(이벤트)에서 파생되며 언제든 재구성할 수 있습니다. Spice OS는 PostgreSQL, Elasticsearch, Redis를 리드 모델로 사용합니다.

**RID (Resource Identifier)**
플랫폼 내 어떤 리소스든 전역적으로 유일하게 식별하는 ID입니다. 형식은 `ri.{service}.{namespace}.{type}.{id}` 입니다. 예: `ri.ontology.main.object.emp-042`.

**SearchJsonQueryV2**
v2 Search API에서 사용하는 쿼리 형식입니다. 쿼리는 연산자를 식별하는 `type` 필드를 가진 JSON 객체입니다(`eq`, `gt`, `gte`, `lt`, `lte`, `isNull`, `containsAnyTerm`, `containsAllTerms`, `containsAllTermsInOrder`, `containsAllTermsInOrderPrefixLastTerm`, `startsWith`, `and`, `or`, `not`).

**Snapshot**
특정 시퀀스 번호에서 aggregate 상태를 캡처한 시점 스냅샷입니다. 스냅샷은 이벤트 리플레이를 가속하여, 스냅샷 이전 이벤트를 건너뛸 수 있게 합니다.

**Transform**
파이프라인에서 데이터를 처리하는 단일 단계입니다. 각 transform은 타입(예: `map`, `filter`, `lookup`)과 설정 파라미터를 가지며 순차적으로 실행됩니다.

**UserPrincipal**
인증 성공 후 `request.state.user`에 부착되는 frozen dataclass입니다. `id`, `type`, `email`, `roles`, `tenant_id`, `org_id`, `verified`, `claims`를 포함합니다. `scopes()` 메서드는 claims dict에서 OAuth scope를 추출합니다. [인증 플로우 가이드](/docs/guides/integration-developer/auth-flow)를 참고하세요.
