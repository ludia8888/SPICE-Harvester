# 아키텍처 투어

Spice OS 아키텍처를 함께 둘러보는 가이드예요.
시스템이 어떻게 구성되어 있는지, 데이터가 어떤 경로로 흐르는지, 서비스끼리 어떻게 협력하는지를 순서대로 살펴봅니다.

## 큰 그림

Spice OS는 총 **26개의 Docker 서비스**로 이루어진 분산 시스템이에요.

### 레이어별 역할

각 레이어가 무엇을 담당하는지 간단히 정리하면 이래요.

| 레이어 | 하는 일 |
|-------|---------------|
| **API Layer** | HTTP 요청을 받고, 인증하고, 적절한 곳으로 라우팅해요 |
| **Core Services** | 비즈니스 로직 실행, 스키마 관리, 검색을 처리합니다 |
| **Worker Layer** | Kafka 이벤트를 소비하며 백그라운드 작업을 수행해요 |
| **Infrastructure** | 데이터 저장, 메시징, 캐싱, 버저닝 등 기반 인프라입니다 |

## 요청 여정: 객체 읽기

실제 API 호출이 내부에서 어떻게 처리되는지 따라가 볼까요?
`GET /api/v2/ontologies/acme/objectTypes/Employee/objects/EMP-042` 요청을 예로 들어 볼게요.

```
1. Client sends HTTP GET
   │
2. BFF Auth Middleware
   ├── Extract Bearer token
   ├── Validate against expected tokens
   └── Attach UserPrincipal to request
   │
3. BFF Route Handler (foundry_ontology_v2.py)
   ├── Parse ontology API name → database name
   ├── Parse object type and primary key
   └── Forward to OMS
   │
4. OMS Route Handler
   ├── Query Elasticsearch for the object
   ├── Overlay key specs from PostgreSQL
   └── Return formatted response
   │
5. BFF Response Formatting
   ├── Convert to Foundry v2 format
   ├── Add __rid, __primaryKey, __apiName
   └── Return to client
```

**거치는 서비스:** BFF → OMS → Elasticsearch (+ PostgreSQL에서 key specs 조회)

**일반적인 응답 시간:** 10-50ms

## 쓰기 경로: 객체 생성

이번에는 쓰기 요청이에요.
`POST /api/v2/ontologies/acme/actions/createObject` 요청이 어떻게 처리되는지 살펴볼게요.

```
1. Client sends action request
   │
2. BFF Auth + Validation
   ├── Authenticate user
   ├── Validate action parameters
   └── Check permissions
   │
3. BFF Action Handler
   ├── Build action command
   ├── Generate idempotency key
   └── Submit to action pipeline
   │
4. Action Processing
   ├── Validate against ontology schema
   ├── Create immutable event
   └── Persist to event store (S3/MinIO)
   │
5. Event Distribution (Kafka)
   ├── Publish event to Kafka topic
   └── Multiple consumers receive event
   │
6. Worker Processing (parallel)
   │
   ├── Instance Worker
   │   └── Write object to PostgreSQL
   │
   ├── Elasticsearch Indexer
   │   └── Index object for search
   │
   ├── Projection Worker
   │   └── Update aggregated views
   │
   └── Lineage Worker
       └── Record provenance edges
```

**거치는 서비스:** BFF → OMS → Kafka → (Instance Worker → PostgreSQL, ES Indexer → Elasticsearch, Projection Worker → PostgreSQL)

**일반적인 응답 시간:** 응답(승인)까지 약 100ms, 모든 읽기 모델에 완전히 반영되기까지는 1-5초 정도 걸려요

## 검색 경로: 객체 찾기

검색은 읽기 전용 경로라서 비교적 단순한 편이에요.
`POST /api/v2/ontologies/acme/objectTypes/Employee/objects/search` 요청을 따라가 볼게요.

```
1. Client sends search query
   │
2. BFF Route Handler
   ├── Parse SearchJsonQueryV2 query
   ├── Validate operators and nesting depth (max 3)
   └── Forward to OMS
   │
3. OMS Search Handler
   ├── Convert to Elasticsearch DSL
   ├── Apply tenant scoping
   ├── Execute against Elasticsearch
   └── Format results with __rid, __primaryKey
   │
4. BFF Response
   ├── Add pagination token
   └── Return to client
```

**거치는 서비스:** BFF → OMS → Elasticsearch

**일반적인 응답 시간:** 20-100ms

## 데이터 임포트 경로

외부 데이터베이스에서 테이블을 가져오는 과정이에요.
임포트가 실행되면 내부에서 어떤 일이 벌어지는지 살펴볼게요.

```
1. Execute import request
   │
2. BFF Connectivity Handler
   ├── Load connection config
   ├── Decrypt secrets (AES-GCM)
   └── Select connector adapter
   │
3. Connector Adapter (e.g., PostgreSQL)
   ├── SQL Query Guard (normalize, inject check)
   ├── Execute query (300s timeout)
   └── Return result set
   │
4. Sync Mode Processing
   ├── SNAPSHOT: Replace all data
   ├── APPEND: Deduplicate and append
   ├── UPDATE: Upsert by primary key
   ├── INCREMENTAL: Watermark-based fetch
   └── CDC: Change token tracking
   │
5. Storage
   ├── Write to lakeFS (versioned commit)
   ├── Register in Dataset Registry
   └── Publish completion event to Kafka
```

**거치는 서비스:** BFF → 외부 DB → LakeFS → MinIO → PostgreSQL → Kafka

## 파이프라인 실행 경로

파이프라인 빌드가 트리거되면 어떤 순서로 진행되는지 볼까요?

```
1. Pipeline build request
   │
2. Pipeline Worker receives job (from Kafka)
   ├── Load pipeline definition
   ├── Normalize nodes and edges
   └── Topological sort (DAG order)
   │
3. Execute nodes in order
   │
   ├── Input Node
   │   └── Load data from dataset (LakeFS)
   │
   ├── Transform Node (38 types)
   │   ├── filter, compute, join, aggregate...
   │   ├── UDF (sandboxed Python)
   │   └── Schema validation at each step
   │
   └── Output Node
       ├── Validate schema contract
       ├── Check expectations (unique, required, FK)
       └── Write to dataset / ontology
   │
4. Post-processing
   ├── Record build artifacts in S3
   ├── Update pipeline registry
   └── Publish lineage edges
```

## 이벤트 라이프사이클

Spice OS에서는 모든 상태 변경이 이벤트로 기록되거든요.
하나의 이벤트가 생성되어 소비되기까지의 흐름은 다음과 같아요.

```
Action Submitted
      │
      ▼
┌─────────────┐
│  Validated   │  Schema check, permission check
└──────┬──────┘
       │
       ▼
┌─────────────┐
│ Event Created│  Immutable, timestamped, actor-tagged
└──────┬──────┘
       │
       ├──────────────────┐
       ▼                  ▼
┌─────────────┐    ┌─────────────┐
│  Kafka Topic│    │ Event Store │  S3/MinIO (durable)
│  (real-time)│    │ (archive)   │
└──────┬──────┘    └─────────────┘
       │
       ├─────────┬──────────┬──────────┐
       ▼         ▼          ▼          ▼
  Instance   Projection  Lineage   Action
  Worker     Worker      Worker    Outbox
       │         │          │          │
       ▼         ▼          ▼          ▼
  PostgreSQL  PostgreSQL  Event     Kafka
  (objects)   (views)     Store    (downstream)
       │
       ▼
  Elasticsearch
  (search index)
```

## 서비스 통신 맵

어떤 서비스가 어디에 어떤 프로토콜로 통신하는지 한눈에 정리한 표예요.

| 출발 | 도착 | 프로토콜 | 목적 |
|------|------|---------|---------|
| Client | BFF | HTTP | API 요청 |
| BFF | OMS | HTTP | 스키마 조회, 검색 처리 |
| BFF | Kafka | Kafka | 이벤트 발행 |
| BFF | Redis | Redis | 캐싱, 레이트 리미팅 |
| BFF | PostgreSQL | SQL | 설정값, 레지스트리 조회 |
| BFF | LakeFS | HTTP | 데이터셋 저장 |
| Workers | Kafka | Kafka | 이벤트 소비 |
| Workers | PostgreSQL | SQL | 읽기 모델 업데이트 |
| Workers | Elasticsearch | HTTP | 검색 인덱스 갱신 |
| Workers | MinIO | S3 | 파일 저장 |
| OMS | PostgreSQL | SQL | 스키마 저장 |
| OMS | Elasticsearch | HTTP | 검색 쿼리 실행 |

## 직접 탐색해 보기

이론만으로는 감이 잘 안 올 수 있으니, 직접 명령어를 실행하면서 확인해 보세요.

### 모든 서비스가 실행 중인지 확인

```bash
docker compose ps
```

모든 서비스가 `Up (healthy)`로 표시되면 정상이에요.

### BFF 헬스 확인

```bash
curl http://localhost:8080/api/health | jq
```

### API 탐색

```bash
# 온톨로지 목록
curl -H "Authorization: Bearer YOUR_TOKEN" \
  http://localhost:8080/api/v2/ontologies | jq

# 객체 유형 목록
curl -H "Authorization: Bearer YOUR_TOKEN" \
  http://localhost:8080/api/v2/ontologies/acme/objectTypes | jq
```

### Kafka 토픽 모니터링

```bash
docker compose exec kafka kafka-topics \
  --bootstrap-server localhost:29092 --list
```

### MinIO 스토리지 보기

브라우저에서 http://localhost:9001 을 열고 MinIO 자격 증명으로 로그인하세요.

### lakeFS 보기

브라우저에서 http://localhost:48080 을 열면 데이터 버저닝 리포지토리를 확인할 수 있어요.

### Elasticsearch 인덱스 확인

```bash
curl http://localhost:9200/_cat/indices?v
```

## 주요 설계 패턴

Spice OS 곳곳에서 반복적으로 쓰이는 설계 패턴을 정리했어요. 코드를 읽을 때 참고하면 흐름이 더 잘 보일 거예요.

| 패턴 | 적용 위치 | 왜 쓰나요? |
|---------|-----------|-----|
| **Event Sourcing** | 모든 쓰기 작업 | 감사 추적, 되돌리기, 상태 리빌드가 가능해요 |
| **CQRS** | BFF(쓰기) + OMS(읽기) | 읽기와 쓰기 경로를 분리해 각각 최적화합니다 |
| **Outbox Pattern** | Action outbox worker | 최소 한 번(at-least-once) 전달을 보장해요 |
| **Adapter Pattern** | 커넥터 어댑터 | 데이터 소스를 플러그 방식으로 추가할 수 있어요 |
| **DAG Execution** | 파이프라인 실행기 | 변환 순서를 결정적으로 보장합니다 |
| **Sandbox** | UDF 런타임 | 사용자 코드를 안전하게 격리 실행해요 |
