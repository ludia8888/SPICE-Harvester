# 릴리스 프로세스

Spice OS 릴리스가 어떻게 버전 관리되고, 빌드와 테스트를 거쳐 배포되는지 알아볼까요.

## 버저닝(Versioning)

Spice OS는 **시맨틱 버저닝(SemVer)** 을 따릅니다:

```
MAJOR.MINOR.PATCH
```

| 구성 요소 | 증가 시점 |
|-----------|---------------|
| **MAJOR** | 브레이킹 API 변경, 호환되지 않는 스키마 마이그레이션, v1/v2 API 제거 |
| **MINOR** | 신규 기능, 신규 API 엔드포인트, 하위 호환 추가 |
| **PATCH** | 버그 수정, 보안 패치, 문서 업데이트 |

프리릴리스 버전은 접미사를 사용합니다: `1.2.0-alpha.1`, `1.2.0-beta.1`, `1.2.0-rc.1`.

## 릴리스 워크플로

### 1. 릴리스 브랜치 준비

```bash
# main에서 릴리스 브랜치 생성
git checkout main
git pull origin main
git checkout -b release/1.2.0
```

### 2. 버전 범프

코드베이스 전반의 버전 번호를 업데이트합니다:

```bash
# bump 스크립트로 모든 버전 레퍼런스 갱신
python scripts/bump_version.py 1.2.0
```

이 스크립트는 다음을 업데이트합니다:
- `pyproject.toml` -- Python 패키지 버전
- `package.json` -- Node.js 패키지 버전
- `src/spice/__version__.py` -- 런타임 버전 상수
- `docker-compose.yml` -- 이미지 태그
- `docs/` -- 문서 내 버전 레퍼런스

### 3. 변경 로그(Changelog) 업데이트

Keep a Changelog 형식으로 `CHANGELOG.md`에 새 섹션을 추가합니다:

```markdown
## [1.2.0] - 2024-01-15

### Added
- Search operator `startsWith` for keyword fields (#1234)
- Pipeline transform `pivot` for row-to-column conversion (#1250)

### Changed
- Increased default `pageSize` from 50 to 100 (#1260)

### Fixed
- Race condition in concurrent action validation (#1245)
- Elasticsearch index mapping for geopoint fields (#1248)

### Deprecated
- v1 API endpoints (scheduled for removal in 2.0.0)
```

### 4. 전체 테스트 스위트 실행

```bash
# 유닛 테스트
pytest -m "not integration"

# 통합 테스트
pytest -m integration

# TypeScript 테스트
npm test

# 타입 체크
mypy src/

# 린트
ruff check .
npx eslint .

# Docker 이미지 빌드
docker compose build
```

### 5. Pull Request 생성

`release/1.2.0`에서 `main`으로 PR을 엽니다. PR에는 다음이 포함되어야 합니다:

- 버전 범프 커밋
- 변경 로그 업데이트
- 마지막 순간의 수정 사항(필요 시)

CI 파이프라인은 다음을 수행합니다:
- 모든 테스트(unit/integration/end-to-end)
- 린팅과 타입 체크
- Docker 이미지 빌드 검증
- 문서 빌드

### 6. 머지 및 태깅

PR이 승인되고 CI가 통과하면:

```bash
# 릴리스 브랜치 머지
git checkout main
git merge release/1.2.0

# 서명된 태그 생성
git tag -s v1.2.0 -m "Release v1.2.0"

# 푸시
git push origin main
git push origin v1.2.0
```

### 7. 빌드 및 퍼블리시

태그 푸시 시 CI/CD 파이프라인이 자동 트리거됩니다:

1. **Docker 이미지 빌드** -- 모든 서비스 이미지를 버전 태그로 빌드
2. **레지스트리에 푸시** -- 컨테이너 레지스트리로 이미지 푸시
3. **Python 패키지 배포** -- SDK를 PyPI에 배포(해당 시)
4. **npm 패키지 배포** -- TypeScript SDK를 npm에 배포(해당 시)
5. **GitHub Release 생성** -- 변경 로그를 본문으로 릴리스를 생성

### 8. 배포

배포는 대상 환경에 따라 달라집니다:

#### 스테이징

```bash
# 최신 이미지 풀
docker compose pull

# 마이그레이션 적용
docker compose exec oms python -m alembic upgrade head

# 새 이미지로 서비스 재시작
docker compose up -d
```

#### 프로덕션

프로덕션 배포는 롤링 업데이트 전략을 사용합니다:

1. (있는 경우) 데이터 스토어 마이그레이션을 먼저 적용
2. 워커 배포(무상태이므로 안전하게 교체 가능)
3. 로드밸런서 뒤에서 API 서버를 하나씩 배포
4. 각 서비스의 헬스 체크를 확인한 뒤 다음 단계 진행
5. 롤아웃 중 에러율 스파이크가 없는지 대시보드 모니터링

## 배포 파이프라인

```
Tag Push ──▶ CI/CD Pipeline
              ├── Build Docker images
              ├── Run full test suite
              ├── Push images to registry
              ├── Deploy to staging
              ├── Run smoke tests
              └── (Manual gate) Deploy to production
```

## 핫픽스 프로세스

다음 계획된 릴리스를 기다릴 수 없는 긴급 수정의 경우:

```bash
# 최신 릴리스 태그에서 핫픽스 브랜치 생성
git checkout -b hotfix/1.2.1 v1.2.0

# 수정 적용
# ... make changes ...

# patch 버전 증가
python scripts/bump_version.py 1.2.1

# 변경 로그 업데이트
# ... add hotfix entry ...

# PR 생성, 머지, 태그, 배포
git tag -s v1.2.1 -m "Hotfix v1.2.1"
git push origin v1.2.1
```

## 롤백(Rollback)

릴리스가 프로덕션에서 문제를 일으키는 경우:

### 빠른 롤백(Quick Rollback)

```bash
# 이전 버전으로 롤백
docker compose pull   # with previous version tag
docker compose up -d
```

### 데이터베이스 롤백

릴리스에 DB 마이그레이션이 포함되었다면:

```bash
# 이전 마이그레이션으로 다운그레이드
docker compose exec oms python -m alembic downgrade -1
```

:::caution
마이그레이션 이후에 데이터가 들어간 컬럼/테이블을 롤백하면 데이터 손실이 발생할 수 있습니다. 롤백은 항상 스테이징에서 먼저 테스트하세요.
:::

## 릴리스 체크리스트

- [ ] 모든 파일의 버전이 업데이트됨
- [ ] 변경 로그가 업데이트됨
- [ ] 모든 테스트 통과(unit/integration/e2e)
- [ ] 린트/타입 체크 경고 없음
- [ ] Docker 이미지 빌드 성공
- [ ] API 변경 사항에 대한 문서 업데이트
- [ ] 브레이킹 변경이 문서화되고 공유됨
- [ ] 릴리스 PR이 최소 2명의 리뷰어에게 승인됨
- [ ] 태그가 생성되고 푸시됨
- [ ] CI/CD 파이프라인이 성공적으로 완료됨
- [ ] 스테이징 배포가 검증됨
- [ ] 프로덕션 배포가 검증됨
- [ ] 릴리스 후 모니터링(최소 30분)
