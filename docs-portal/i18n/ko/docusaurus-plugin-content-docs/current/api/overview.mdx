# API 개요

Spice OS는 `/api/v2/ontologies/*` 경로에 Foundry 호환 공개 API를, `/api/v1/*` 경로에 내부 호환 라우트를 제공합니다.
이 페이지에서는 인증, 기본 URL, 버전 관리, 페이지네이션, 에러 처리 규칙을 살펴볼까요.

## 기본 URL(Base URL)

모든 API 요청은 BFF 게이트웨이를 거쳐 전달돼요.

```
https://your-domain.com/api/{version}/
```

로컬 개발 환경:

```
http://localhost:8002/api/{version}/
```

## 인증(Authentication)

모든 API 요청에는 `Authorization` 헤더에 Bearer 토큰을 포함해야 해요.

```bash
curl -H "Authorization: Bearer <access_token>" \
  https://your-domain.com/api/v2/ontologies
```

### 토큰 발급

인증 엔드포인트에 토큰을 요청하면 됩니다.

```bash
curl -X POST https://your-domain.com/api/v2/auth/token \
  -H "Content-Type: application/json" \
  -d '{
    "username": "your-username",
    "password": "your-password"
  }'
```

응답:

```json
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 3600
}
```

토큰은 기본 1시간 뒤에 만료돼요. 만료 전에 refresh 플로우를 사용하거나, 다시 인증해 주세요.

### 토큰 갱신(Token Refresh)

```bash
curl -X POST https://your-domain.com/api/v2/auth/refresh \
  -H "Content-Type: application/json" \
  -d '{
    "refresh_token": "your-refresh-token"
  }'
```

## API 버전 관리(API Versioning)

Spice OS API는 계약(Contract) 역할에 따라 두 가지 버전으로 나뉘어요.

| 버전 | 계약 역할 | 상태 | 설명 |
|---------|---------------|--------|-------------|
| **v2** | Foundry 호환 공개 API | **Active** | 외부 및 신규 연동에 권장 |
| **v1** | 내부 호환/운영용 API | Restricted | Foundry 공개 온톨로지 계약이 아님 |

### 버전 선택

URL 경로에 버전을 포함하면 돼요.

```
/api/v2/ontologies/...    # Foundry 호환 공개 API (권장)
/api/v1/...               # 내부 호환/운영용 API
```

### v1 → v2 마이그레이션

v2 API는 Foundry 호환 정식 통합 계약이에요. 주요 차이점을 정리하면 다음과 같아요.

| 항목 | v1 | v2 |
|--------|----|----|
| URL 구조 | 제품/내부 중심 | `/api/v2/ontologies/{ontology}/...` |
| 계약 범위 | 호환성 라우트 혼용 | Foundry 호환 공개 온톨로지 리소스 |
| 검색(Search) | 쿼리 형태 혼용 | `SearchJsonQueryV2` 통일 |
| 페이지네이션 | offset/token 혼용 | opaque `pageToken` 통일 |
| 에러 포맷 | 엔벨로프 혼용 | `{errorCode, errorName, errorInstanceId, parameters}` 통일 |

단계별 마이그레이션 방법은 [Foundry 마이그레이션 가이드](/docs/guides/platform-admin/foundry-migration)를 참고해 주세요.

## 페이지네이션(Pagination)

v2 API는 **opaque page token** 방식의 페이지네이션을 사용해요. 데이터가 변경되더라도 안정적으로 페이지를 탐색할 수 있거든요.

### 요청(Request)

```bash
# 첫 페이지
curl "https://your-domain.com/api/v2/ontologies/acme/objects/Employee?pageSize=25"

# 다음 페이지
curl "https://your-domain.com/api/v2/ontologies/acme/objects/Employee?pageSize=25&pageToken=eyJvZmZzZXQi..."
```

### 응답(Response)

```json
{
  "data": [ ... ],
  "nextPageToken": "eyJvZmZzZXQiOjI1fQ=="
}
```

- 더 이상 결과가 없으면 `nextPageToken`이 `null`로 돌아와요.
- page token은 불투명(opaque) 문자열이에요. 직접 파싱하거나 조합하지 마세요.
- 기본 `pageSize`는 100이고, 최대 1000까지 설정할 수 있어요.
- token은 10분 동안 사용하지 않으면 만료돼요.

## 요청 형식(Request Format)

### 헤더(Headers)

| 헤더 | 필수 | 설명 |
|--------|----------|-------------|
| `Authorization` | Yes | `Bearer <access_token>` |
| `Content-Type` | POST/PUT에서 필요 | `application/json` |
| `Accept` | No | `application/json` (기본) |
| `X-Request-Id` | No | 트레이싱을 위한 클라이언트 제공 request ID |
| `X-Consistency` | No | read-after-write 정합성을 위해 `strong` |

### 콘텐츠 타입(Content Types)

- 요청 바디는 JSON(`application/json`)이어야 해요.
- 파일 업로드에는 `multipart/form-data`를 사용합니다.
- 응답은 항상 JSON 형식이에요.

## 응답 형식(Response Format)

### 성공 응답(Success Responses)

단일 객체:

```json
{
  "__rid": "ri.ontology.main.object.emp-042",
  "__primaryKey": "EMP-042",
  "__apiName": "Employee",
  "properties": {
    "employeeId": "EMP-042",
    "fullName": "Jane Doe"
  }
}
```

컬렉션:

```json
{
  "data": [
    { "__rid": "...", "__primaryKey": "...", ... },
    { "__rid": "...", "__primaryKey": "...", ... }
  ],
  "nextPageToken": "eyJvZmZzZXQi..."
}
```

### 에러 응답(FoundryAPIError)

모든 `/api/v2/` 엔드포인트는 **FoundryAPIError** 엔벨로프로 에러를 반환해요. BFF 서비스에 등록된 전역 `foundry_exception_handler`가 이 Foundry 호환 포맷을 처리하거든요. v2 경로의 요청 검증 오류나 JSON 디코딩 오류도 자동으로 같은 포맷으로 래핑됩니다.

```json
{
  "errorCode": "NOT_FOUND",
  "errorName": "ObjectNotFound",
  "errorInstanceId": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "parameters": {
    "objectType": "Employee",
    "primaryKey": "EMP-999"
  }
}
```

| 필드 | 타입 | 설명 |
|-------|------|-------------|
| `errorCode` | string | 기계가 읽을 수 있는 오류 코드 (예: `NOT_FOUND`, `PERMISSION_DENIED`) |
| `errorName` | string | 사람이 읽기 좋은 오류 이름 (예: `ObjectNotFound`, `ApiUsageDenied`) |
| `errorInstanceId` | string | 오류 발생마다 생성되는 UUID -- 트레이싱에 활용 |
| `parameters` | object | 디버깅용 컨텍스트 파라미터 |

### HTTP 상태 코드

| 코드 | 의미 | 사용 시점 |
|------|---------|-----------|
| `200` | OK | 읽기/변이 성공 |
| `201` | Created | 객체 생성 성공 |
| `204` | No Content | 삭제 성공 |
| `400` | Bad Request | 잘못된 바디/파라미터 |
| `401` | Unauthorized | Bearer 토큰 누락/무효 |
| `403` | Forbidden | 권한 부족 |
| `404` | Not Found | 리소스 없음 |
| `409` | Conflict | 동시 수정 충돌 |
| `422` | Unprocessable Entity | 검증 실패 |
| `429` | Too Many Requests | 레이트 리미트 초과 |
| `500` | Internal Server Error | 예상치 못한 서버 오류 |

## 레이트 리미팅(Rate Limiting)

BFF 게이트웨이는 인증된 사용자 단위로 레이트 리미트를 적용해요.

| 티어 | 분당 요청 | 버스트 |
|------|-----------------|-------|
| Standard | 600 | 100 |
| Premium | 3000 | 500 |
| Admin | Unlimited | Unlimited |

레이트 리미트에 도달하면 아래와 같은 헤더가 응답에 포함돼요.

```
HTTP/1.1 429 Too Many Requests
Retry-After: 30
X-RateLimit-Limit: 600
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1705312200
```

**총 엔드포인트: 264개** (v2 109개 + v1 155개)
