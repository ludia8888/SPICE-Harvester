# 오류 코드 레퍼런스

Spice OS는 모든 서브시스템에 걸쳐 290개 이상의 구조화된 오류 코드를 정의해요. 오류 형식, 네이밍 컨벤션, severity 레벨, 재시도 정책, 자주 발생하는 오류 코드를 한눈에 확인할 수 있습니다.

## 오류 응답 형식(Error Response Format)

모든 API 오류는 구조화된 JSON 응답을 반환합니다:

```json
{
  "errorCode": "NOT_FOUND",
  "errorName": "ObjectNotFound",
  "errorInstanceId": "err-2024-01-15-a1b2c3d4",
  "parameters": {
    "objectType": "Employee",
    "primaryKey": "EMP-999"
  }
}
```

| 필드 | 타입 | 설명 |
|-------|------|-------------|
| `errorCode` | string | 분류 체계를 따르는 머신 리더블 코드 |
| `errorName` | string | PascalCase의 사람이 읽기 좋은 오류 이름 |
| `errorInstanceId` | string | 이 오류 발생의 고유 식별자. 지원 요청 시 사용 |
| `parameters` | object | 디버깅을 위한 컨텍스트 정보(오류 타입에 따라 다름) |

## 오류 코드 구조(Error Code Structure)

오류 코드는 다음 패턴을 따릅니다:

```
SHV-{SUBSYSTEM}-{DOMAIN}-{CLASS}-{NUMBER}
```

| 세그먼트 | 설명 | 예시 |
|---------|-------------|---------|
| `SHV` | 플랫폼 접두사(Spice OS) | 항상 `SHV` |
| `SUBSYSTEM` | 서비스 서브시스템 | `ONT`, `ACT`, `SRH`, `PIP`, `LIN`, `AUTH`, `SYS` |
| `DOMAIN` | 기능 도메인 | `OBJ`, `SCH`, `QRY`, `IDX`, `EVT`, `CFG` |
| `CLASS` | 오류 분류 | `VAL`, `NFD`, `CON`, `PRM`, `INT`, `TMO` |
| `NUMBER` | 순번 | `001`-`999` |

**예시:** `SHV-ONT-OBJ-NFD-001` = Spice OS, Ontology 서브시스템, Object 도메인, Not Found 클래스, 오류 #001.

## 서브시스템(Subsystems)

| 코드 | 서브시스템 | 설명 |
|------|-----------|-------------|
| `ONT` | Ontology | 객체 유형, 속성, 링크 유형, 인스턴스 |
| `ACT` | Actions | 액션 검증, 실행, undo, 템플릿 |
| `SRH` | Search | 쿼리 파싱, Elasticsearch, 결과 포맷 |
| `PIP` | Pipeline | 파이프라인 정의/실행/변환 |
| `LIN` | Lineage | 데이터 라인리지 추적 및 조회 |
| `AUTH` | Authentication | 토큰 관리, 인가, RBAC |
| `SYS` | System | 인프라, 설정, 헬스 |
| `EVT` | Events | 이벤트 스토어, Kafka, 이벤트 처리 |
| `PRJ` | Projections | 프로젝션 빌드 및 정합성 |
| `ING` | Ingestion | 커넥터 인제스트, 파일 업로드, 대량 임포트, 타입 추론 런타임 |

## 도메인(Domains)

| 코드 | 도메인 | 설명 |
|------|--------|-------------|
| `OBJ` | Object | 객체 인스턴스 작업 |
| `SCH` | Schema | 객체 유형 및 속성 정의 |
| `QRY` | Query | 검색/필터 작업 |
| `IDX` | Index | Elasticsearch 인덱싱 |
| `EVT` | Event | 이벤트 생성/처리 |
| `CFG` | Config | 설정 및 구성 |
| `NET` | Network | 서비스 간 통신 |
| `STR` | Storage | DB 및 파일 저장 |
| `TRN` | Transform | 파이프라인 transform 실행 |
| `TPL` | Template | 액션 템플릿 평가 |

## 오류 클래스(Error Classes)

| 코드 | 클래스 | HTTP 상태 | 재시도 가능 | 설명 |
|------|-------|-------------|-----------|-------------|
| `VAL` | Validation | 400 / 422 | No | 입력 검증 실패 |
| `NFD` | Not Found | 404 | No | 리소스가 존재하지 않음 |
| `CON` | Conflict | 409 | Yes (re-read 후) | 동시 수정 충돌 |
| `PRM` | Permission | 403 | No | 권한 부족 |
| `INT` | Internal | 500 | Yes | 예상치 못한 내부 오류 |
| `TMO` | Timeout | 504 | Yes | 작업 타임아웃 |
| `LIM` | Rate Limit | 429 | Yes (지연 후) | 레이트 리미트 초과 |
| `UNV` | Unavailable | 503 | Yes | 서비스 일시적 사용 불가 |
| `DUP` | Duplicate | 409 | No | 중복 리소스 |
| `DEP` | Dependency | 424 | Yes | 하위 의존성 실패 |

## 심각도 레벨(Severity Levels)

각 오류는 로깅/알림 동작을 결정하는 severity 레벨을 가집니다:

| 레벨 | 설명 | 알림 |
|-------|-------------|----------|
| `INFO` | 정상 사용 중 기대 가능한 오류(예: object not found) | 없음 |
| `WARNING` | 문제 가능성을 시사하는 비정상 상태(예: 레이트 리미트 임박) | warning 로그 |
| `ERROR` | 요청 완료를 막는 실패 | 지속 발생 시 알림 |
| `CRITICAL` | 즉각 대응이 필요한 시스템 레벨 실패 | 즉시 페이지 |

## 재시도 정책(Retry Policies)

### 일반 재시도 전략

재시도 가능한 오류에 대해서는 지터(jitter)가 있는 지수 백오프를 사용하세요:

```
delay = min(base_delay * 2^attempt + random_jitter, max_delay)
```

| 파라미터 | 기본값 | 설명 |
|-----------|---------|-------------|
| `base_delay` | 100 ms | 초기 재시도 지연 |
| `max_delay` | 30 s | 재시도 간 최대 지연 |
| `max_attempts` | 3 | 최대 재시도 횟수 |
| `jitter` | 0-100 ms | thundering herd 방지를 위한 랜덤 지터 |

### 클래스별 재시도 정책

| 오류 클래스 | 재시도 가능 | 전략 | 비고 |
|-------------|-----------|----------|-------|
| `VAL` | No | -- | 요청 payload를 수정 |
| `NFD` | No | -- | 리소스 식별자 확인 |
| `CON` | Yes | re-read + retry | 최신 버전을 읽고 재제출 |
| `PRM` | No | -- | 사용자 권한 확인 |
| `INT` | Yes | 지수 백오프 | 대기 후 재시도 |
| `TMO` | Yes | 지수 백오프 | 요청 범위 축소 고려 |
| `LIM` | Yes | `Retry-After` 준수 | 지정 시간 대기 |
| `UNV` | Yes | 지수 백오프 | 서비스 일시 다운 |
| `DUP` | No | -- | 리소스가 이미 존재 |
| `DEP` | Yes | 지수 백오프 | 하위 의존성 이슈 |

## 자주 발생하는 오류 코드(Common Error Codes)

### Ontology 오류

| 코드 | 이름 | HTTP | 설명 |
|------|------|------|-------------|
| `SHV-ONT-OBJ-NFD-001` | `ObjectNotFound` | 404 | 기본키에 해당하는 객체가 존재하지 않음 |
| `SHV-ONT-OBJ-VAL-001` | `InvalidObjectProperties` | 422 | 객체 속성이 스키마 검증에 실패 |
| `SHV-ONT-OBJ-CON-001` | `ConcurrentModification` | 409 | 객체가 다른 요청에 의해 수정됨 |
| `SHV-ONT-OBJ-DUP-001` | `DuplicatePrimaryKey` | 409 | 동일 기본키의 객체가 이미 존재 |
| `SHV-ONT-SCH-NFD-001` | `ObjectTypeNotFound` | 404 | 온톨로지에 객체 유형이 존재하지 않음 |
| `SHV-ONT-SCH-NFD-002` | `OntologyNotFound` | 404 | 온톨로지가 존재하지 않음 |
| `SHV-ONT-SCH-VAL-001` | `InvalidPropertyDefinition` | 422 | 속성 정의가 유효하지 않음 |
| `SHV-ONT-SCH-VAL-002` | `InvalidPrimaryKey` | 422 | 기본키 속성이 누락되었거나 유효하지 않음 |

## 클라이언트 코드에서 오류 처리(Handling Errors in Client Code)

### Python 예시

```python
import requests

resp = requests.get(
    "http://localhost:8002/api/v2/ontologies/demo/objects/Employee/EMP-999",
    headers={"Authorization": "Bearer YOUR_TOKEN"},
)

if resp.status_code >= 400:
    err = resp.json()
    print("errorCode:", err.get("errorCode"))
    print("errorName:", err.get("errorName"))
    print("errorInstanceId:", err.get("errorInstanceId"))
    print("parameters:", err.get("parameters"))
```

### TypeScript 예시

```ts
const resp = await fetch(
  "http://localhost:8002/api/v2/ontologies/demo/objects/Employee/EMP-999",
  { headers: { Authorization: "Bearer YOUR_TOKEN" } },
);

if (!resp.ok) {
  const err = await resp.json();
  console.log("errorCode:", err.errorCode);
  console.log("errorName:", err.errorName);
  console.log("errorInstanceId:", err.errorInstanceId);
  console.log("parameters:", err.parameters);
}
```
