백엔드 개발 요구사항서 
모든 요구사항에 대한 진척사항은 docs/AGENT_PRD_CHECKLIST.md 여기다가 기록하고 구현사항과 목표사이의 GAP을 기록한다. 
요구 충족시 가능한 유저시나리오:
멀티스텝 작업을 자연스럽게 수행하도록 설계되어 있다. 한 번의 사용자 요청에 대해 여러 개의 툴을 연속적으로 호출하며 목표를 달성할 수 있는데, 앞서 언급한 closed-loop 아키텍처가 이를 가능케 한다 . 예컨데 “새 온톨로지 객체 타입을 만들고 여기에 최근 데이터셋을 연결해줘”라는 사용자 명령이 들어오면, 내부적으로 ① 객체 타입 생성 액션 툴 호출 → ② 해당 데이터셋을 찾기 위한 쿼리 툴 호출 → ③ 객체 타입에 데이터셋 연계(예: 인터페이스 설정) 액션 호출 등의 일련의 도구 실행을 자동으로 계획한다. LLM은 각 단계 실행 후 결과를 확인하여 다음 단계로 피드백받기 때문에, 만약 중간에 실패나 예상치 못한 결과가 나오면 그에 맞춰 계획을 조정하거나 추가 질문을 할 수도 있다 . 이러한 도구 체인의 예는 데이터 파이프라인 코드 작성 작업에서도 볼 수 있다. 새로운 데이터셋 빌드를 명령하면, ① 코드 저장소에 새 브랜치 생성, ② 파이프라인 빌더 트랜스폼 코드 작성, ③ 프리뷰 실행으로 코드 검증 및 오류 수정, ④ 데이터셋 빌드 실행 등의 순서로 여러 툴이 사용되어 하나의 고차원 작업이 완수된다.

1. 인증·권한·테넌시
	•	AUTH-001 시스템은 모든 에이전트 동작(조회/변경/실행)을 항상 현재 로그인한 사용자 보안 컨텍스트(토큰/세션) 로 수행해야 한다.
	•	AUTH-002 시스템은 모든 툴 실행 요청에 대해 RBAC/ABAC 등 기존 플랫폼 권한 검사를 강제해야 하며, 권한 실패 시 툴 실행을 중단하고 표준화된 오류 코드를 반환해야 한다.
	•	AUTH-003 시스템은 사용자 권한을 절대 상승(escalation)시키지 않아야 하며, 서비스 계정/백엔드 권한으로 사용자 권한을 우회하는 실행 경로를 금지해야 한다.
	•	AUTH-004 시스템은 멀티테넌시를 지원해야 하며, 세션/컨텍스트/로그/아티팩트는 테넌트 경계로 완전 격리되어야 한다.
	•	AUTH-005 시스템은 사용자/조직 단위로 허용된 모델 목록, 허용된 툴 목록, 자동 승인 규칙, 데이터/문서 접근 정책을 중앙 정책으로 적용해야 한다.

⸻

2. 세션·대화 상태 관리
	•	SESS-001 시스템은 “에이전트 세션(Session)”을 생성/조회/종료하는 API를 제공해야 한다.
	•	SESS-002 시스템은 세션 단위로 다음을 저장·조회 가능해야 한다.
	•	메시지(사용자/에이전트/시스템)
	•	툴 호출 요청/응답
	•	승인 요청/결과
	•	컨텍스트 첨부 항목 목록
	•	선택된 모델 및 활성화된 툴 목록
	•	토큰/비용/지연시간 계측 값
	•	SESS-003 시스템은 세션 시작 시 기본 컨텍스트가 없는 “클린 상태” 로 시작해야 하며, 이전 세션/다른 세션의 컨텍스트가 자동 유입되지 않아야 한다.
	•	SESS-004 시스템은 세션 단위로 대화 요약(Summarize) 및 대화 구간 제거(Remove) 를 지원해야 하며, 요약/제거 후에도 감사 추적 가능성을 유지해야 한다(감사 로그는 별도 보존).
	•	SESS-005 시스템은 세션의 상태 머신을 제공해야 한다. 최소 상태: ACTIVE, WAITING_APPROVAL, RUNNING_TOOL, ERROR, COMPLETED, TERMINATED.
	•	SESS-006 시스템은 긴 작업을 지원하기 위해 세션 내 실행을 작업(Job) 단위로 분리하고, Job 상태 조회 API를 제공해야 한다.

⸻

3. LLM 모델 게이트웨이 및 모델 선택
	•	LLM-001 시스템은 멀티 LLM 제공자(예: OpenAI/Anthropic/Google/자체호스팅)를 단일 추상 인터페이스로 호출할 수 있어야 한다.
	•	LLM-002 시스템은 조직 관리자가 사전 승인(allowlist)한 모델만 세션에서 선택 가능해야 한다.
	•	LLM-003 시스템은 세션 단위로 모델을 지정/변경할 수 있어야 하며, 모든 호출은 세션에 고정된 정책(보안/마스킹/툴 허용)을 적용해야 한다.
	•	LLM-004 시스템은 툴 호출 방식으로 다음 2가지를 지원해야 한다.
	•	프롬프트 기반(단일 툴 호출 중심)
	•	네이티브 함수/툴 호출(복수 툴 호출 및 연쇄/병렬 계획 가능)
	•	LLM-005 시스템은 모델별 지원 능력(함수 호출 지원 여부, 컨텍스트 길이 등)을 메타데이터로 관리하고, 미지원 기능 요청 시 자동 폴백(예: 네이티브→프롬프트 기반)을 수행해야 한다.
	•	LLM-006 시스템은 모델 호출에 대해 타임아웃/재시도/서킷브레이커를 적용해야 하며, 재시도는 멱등성을 보장하는 범위에서만 허용해야 한다.

⸻

4. 에이전트 오케스트레이션 (Closed-loop 실행 엔진)
	•	AGT-001 시스템은 “Prompt-to-Action” 흐름을 백엔드에서 오케스트레이션해야 한다:
사용자 입력 → 의도 분석/계획 → 툴 선택/파라미터 생성 → 툴 실행 → 결과 관찰 → 다음 계획(반복) → 최종 응답
	•	AGT-002 시스템은 위 반복 루프를 지원하는 실행 엔진을 제공해야 하며, 각 단계의 입력/출력은 구조화된 이벤트로 기록되어야 한다.
	•	AGT-003 시스템은 에이전트가 멀티스텝 작업을 수행할 수 있도록, 세션 내에서 연속 툴 호출 및 툴 실행 결과 기반 재계획을 지원해야 한다.
	•	AGT-004 시스템은 에이전트가 “자체 검증 단계”를 수행할 수 있도록, 프리뷰/드라이런/검증용 툴 호출 패턴을 지원해야 한다(예: 코드 프리뷰, 빌드 프리뷰, CI 결과 확인).
	•	AGT-005 시스템은 에러 발생 시 다음을 지원해야 한다.
	•	실패 원인(권한/검증/런타임/타임아웃) 분류
	•	부분 성공/부분 실패 처리
	•	재시도 가능 여부 판정
	•	사용자를 위한 추가 질문(Clarification)로 전환
	•	AGT-006 시스템은 동시에 여러 툴 호출이 가능한 모드에서 병렬 실행 제한(동시성 제한) 및 실행 순서 제어(의존성 그래프)를 제공해야 한다.

⸻

5. 툴 프레임워크 (레지스트리·스키마·실행)
	•	TOOL-001 시스템은 툴을 “등록 가능한 플러그인”으로 관리해야 하며, 툴별로 다음 메타데이터를 저장해야 한다.
	•	툴 이름/버전
	•	입력 스키마(JSON Schema 수준)
	•	출력 스키마
	•	위험도 등급(승인 필요 여부 판단용)
	•	실행 타임아웃/리트라이 정책
	•	필요한 권한/리소스 범위 선언
	•	TOOL-002 시스템은 세션 단위로 “활성화된 툴 목록”을 저장해야 하며, 비활성화 툴 호출은 런타임에서 강제 차단해야 한다.
	•	TOOL-003 시스템은 툴 호출마다 tool_run_id를 발급하고, 재시도/중복 호출에서도 동일 작업이 중복 실행되지 않도록 멱등성 키를 지원해야 한다.
	•	TOOL-004 시스템은 툴 실행 결과를 표준화된 형태로 반환해야 한다. 최소 필드: status(success|failed), error_code, error_message, payload, side_effect_summary.
	•	TOOL-005 시스템은 다음 툴 유형을 최소 지원해야 한다.
	•	Action 툴(플랫폼 객체/링크/정의 생성·수정·삭제)
	•	Object Query 툴(객체 검색/필터/집계/링크 추적 조회)
	•	Function 툴(등록 함수 실행 및 버전/태그 지정 실행)
	•	Command 툴(외부 애플리케이션 명령 실행 트리거)
	•	Session Variable Update 툴(세션 변수 설정/갱신)
	•	Clarification 툴(추가 질문 생성/요청)
	•	TOOL-006 시스템은 툴 실행 전/후 훅(hook)을 제공하여, 정책 검사/마스킹/감사 로깅/비용 계측을 일관되게 적용해야 한다.
	•	TOOL-007 시스템은 사이드이펙트가 큰 툴에 대해 “드라이런/프리뷰 모드”를 지원할 수 있어야 한다(지원 가능 시), 프리뷰 결과는 본 실행과 분리 저장되어야 한다.

⸻

6. 승인(Human-in-the-loop) 및 정책 엔진
	•	APR-001 시스템은 툴 실행 요청을 “승인 필요/불필요”로 분류하는 정책 엔진을 제공해야 한다.
	•	APR-002 시스템은 다음 범주에 해당하는 실행에 대해 기본적으로 사전 승인을 요구해야 한다.
	•	기본/프로덕션 브랜치(또는 보호된 환경)에 대한 변경
	•	브랜치 격리 없이 발생하는 변경(새 저장소 생성 등)
	•	대규모/비가역적 사이드이펙트(빌드 실행, 배포 트리거, 대량 삭제 등)
	•	APR-003 시스템은 승인 요청 생성 시 다음 정보를 포함해야 한다.
	•	요청 툴/파라미터 요약
	•	예상 변경 범위(대상 리소스/개수/환경)
	•	위험도 등급
	•	롤백 가능 여부/대체(프리뷰) 제안
	•	APR-004 시스템은 승인 결과로 ALLOW 또는 REJECT를 저장하고, REJECT 시 해당 툴 실행은 절대 수행되지 않아야 한다.
	•	APR-005 시스템은 관리자 정책으로 자동 승인 allowlist 규칙을 지원해야 한다(예: 특정 브랜치/프로젝트/툴에 한해 승인 생략).
	•	APR-006 시스템은 승인 대기 상태에서 세션을 WAITING_APPROVAL로 전환하고, 승인 결과 수신 시 정확히 해당 실행 흐름을 재개해야 한다.

⸻

7. 변경 격리·리뷰 아티팩트(브랜치/프로포절/PR/CI)
	•	CHG-001 시스템은 모든 변경 작업(코드/온톨로지/구성)은 기본적으로 격리된 작업 공간(브랜치/프로포절) 에서 수행되도록 강제해야 한다.
	•	CHG-002 시스템은 변경 결과를 “리뷰 가능한 아티팩트”로 생성해야 한다. 최소 지원:
	•	코드 변경: Pull Request(또는 동등 개념) 생성
	•	온톨로지/메타 변경: Branch Proposal(또는 동등 개념) 생성
	•	CHG-003 시스템은 PR/프로포절에 변경 요약(파일/객체 단위 diff 요약), 생성 원인(세션/요청), 관련 툴 호출 내역을 연결해야 한다.
	•	CHG-004 시스템은 CI/검증 결과를 수집하여 세션에 반환할 수 있어야 하며, 에이전트가 해당 결과를 후속 루프의 입력으로 사용할 수 있어야 한다.
	•	CHG-005 시스템은 “프리뷰 실행(예: 트랜스폼/함수 프리뷰)” 결과를 저장하고, 동일 세션에서 재참조 가능해야 한다.
	•	CHG-006 시스템은 브랜치/프로포절/PR의 링크/식별자를 표준 형태로 제공하여, 프런트에서 사용자에게 연결(탐색) 가능해야 한다.

⸻

8. 컨텍스트 관리 및 RAG(문서/온톨로지/데이터)
	•	CTX-001 시스템은 세션에 첨부 가능한 컨텍스트 항목을 타입별로 관리해야 한다. 최소 타입:
	•	데이터셋(스키마/샘플/메타)
	•	온톨로지 객체 타입/액션 타입/인터페이스 정의
	•	함수(시그니처/설명/버전)
	•	브랜치/프로포절/PR 참조
	•	문서 번들(사내/공식 문서)
	•	파일 업로드(텍스트 추출 결과 포함)
	•	CTX-002 시스템은 “사용자가 명시적으로 첨부한 컨텍스트”만 프롬프트에 포함되도록 강제해야 한다(비첨부 리소스 자동 열람 금지).
	•	CTX-003 시스템은 컨텍스트 항목별로 프롬프트 포함 방식을 지원해야 한다(전량 포함, 요약 포함, 검색 기반 포함 등).
	•	CTX-004 시스템은 문서 번들을 대상으로 한 검색(RAG)을 지원해야 하며, 검색 결과는 출처 식별자(citation용) 와 함께 반환해야 한다.
	•	CTX-005 시스템은 온톨로지/데이터 기반 컨텍스트 추출을 위한 “컨텍스트 툴”(예: 조건 필터링 후 상위 N개 객체 필드 첨부)을 지원해야 한다.
	•	CTX-006 시스템은 파일 업로드에 대해 저장/바이러스 검사/텍스트 추출/크기 제한/보존기간을 적용해야 한다.
	•	CTX-007 시스템은 컨텍스트 윈도우 관리를 위해 항목별 토큰 기여량을 계산·저장하고, 토큰 예산 초과 시 요약/제거 후보를 산출할 수 있어야 한다.

⸻

9. 프롬프트 컴파일·툴 설명 주입·출력 규격화
	•	PRM-001 시스템은 시스템 프롬프트에 다음을 포함하도록 컴파일해야 한다.
	•	에이전트 임무/작동 규칙
	•	세션에서 허용된 툴 목록 및 각 툴 사용법(입력/출력 스키마)
	•	보안/권한/데이터 취급 제한 규칙
	•	승인 필요 시 동작 규칙
	•	PRM-002 시스템은 툴 설명/제약이 세션별 활성화 상태와 정확히 일치하도록 보장해야 한다.
	•	PRM-003 시스템은 에이전트 출력에서 “자연어 응답”과 “툴 호출 계획/요청”을 분리해 처리할 수 있어야 한다(네이티브 호출 또는 명시적 포맷).
	•	PRM-004 시스템은 툴 호출 파라미터에 대해 스키마 검증을 수행하고, 불일치 시 툴 실행을 차단하고 재생성 루프를 유도할 수 있어야 한다.

⸻

10. 감사·투명성·관측(Observability)·비용/토큰 계측
	•	OBS-001 시스템은 모든 툴 호출에 대해 “누가/언제/어떤 세션에서/무슨 입력으로/무슨 결과를 냈는지”를 감사 로그로 남겨야 한다.
	•	OBS-002 시스템은 승인 이벤트(요청/허용/거부), 정책 판정 결과, 실행된 변경 아티팩트(PR/프로포절) 연결 정보를 감사 로그에 포함해야 한다.
	•	OBS-003 시스템은 세션 아웃라인 구현을 위해 다음 이벤트 스트림/조회 API를 제공해야 한다.
	•	메시지 생성 이벤트
	•	툴 호출 시작/종료 이벤트
	•	승인 대기/승인 완료 이벤트
	•	토큰 사용량/비용/지연시간 이벤트
	•	OBS-004 시스템은 메시지/툴 호출 단위로 토큰 사용량을 계산하여 저장해야 하며, 세션 합계 및 모델 최대 컨텍스트 대비 사용량을 제공해야 한다.
	•	OBS-005 시스템은 모델별/사용자별/조직별 비용(또는 과금 단위) 추정치를 집계해야 한다.
	•	OBS-006 시스템은 추적 가능한 분산 트레이싱(trace_id), 상관관계 id(correlation_id)를 모든 요청/툴 실행에 부여해야 한다.
	•	OBS-007 시스템은 운영 지표를 수집해야 한다. 최소: 성공률, 승인 대기 시간, 툴 실행 시간, LLM 응답 시간, 오류 코드 분포, 재시도 횟수.

⸻

11. 프라이버시·데이터 보호·모델 송신 통제
	•	SEC-001 시스템은 LLM 호출 전 송신 데이터에 대해 PII/민감정보 마스킹 정책을 적용할 수 있어야 한다(조직별 정책).
	•	SEC-002 시스템은 LLM 제공자별로 “대화/데이터 저장 금지”, “로그 최소화”, “세션 격리” 등 송신 정책을 구성하고 강제해야 한다.
	•	SEC-003 시스템은 컨텍스트 첨부 데이터에 대해 데이터 분류(기밀 등급) 메타를 존중하고, 금지 등급은 LLM 프롬프트 포함을 차단해야 한다.
	•	SEC-004 시스템은 저장 데이터(세션/로그/업로드 파일)에 대해 암호화(전송/저장) 및 키 관리 정책을 적용해야 한다.
	•	SEC-005 시스템은 데이터 보존기간(retention)을 객체 타입별로 설정 가능해야 하며, 기간 만료 시 자동 삭제/비식별화를 수행해야 한다.

⸻

12. 외부/내부 통합 API 요구사항
	•	INT-001 시스템은 다음 API 군을 제공해야 한다.
	•	세션 생성/조회/종료
	•	메시지 전송 및 응답 스트리밍
	•	컨텍스트 첨부/제거/목록 조회
	•	툴 목록 조회 및 세션 툴 활성화 설정
	•	승인 요청 조회/승인/거부
	•	작업(Job) 상태 조회
	•	아티팩트(PR/프로포절) 링크 조회
	•	INT-002 시스템은 플랫폼 리소스(데이터셋/온톨로지/함수/브랜치) 메타데이터 조회를 위한 커넥터 계층을 제공해야 한다.
	•	INT-003 시스템은 문서 번들 검색을 위한 인덱스/검색 API를 제공해야 하며, 결과는 citation 식별자를 포함해야 한다.
	•	INT-004 시스템은 CI 결과 수집을 위한 통합(웹훅 또는 폴링)과 결과 표준화를 제공해야 한다.

⸻

13. 신뢰성·성능·운영(Non-functional)
	•	NFR-001 시스템은 툴 실행과 LLM 호출에 대해 각각 별도의 타임아웃을 적용해야 하며, 타임아웃 시 세션은 복구 가능한 오류 상태로 전환되어야 한다.
	•	NFR-002 시스템은 장시간 실행 툴을 비동기 Job으로 처리하고, 프런트가 진행률/상태를 조회할 수 있어야 한다.
	•	NFR-003 시스템은 동일 세션 내 동시 실행을 제어해야 하며(락/큐), 메시지·툴 이벤트의 순서 보장을 제공해야 한다.
	•	NFR-004 시스템은 레이트리밋/쿼터를 사용자·조직·모델 단위로 지원해야 한다.
	•	NFR-005 시스템은 장애 격리를 위해 컴포넌트별(LLM/툴 실행/검색/저장소) 폴백 전략을 가져야 한다.
	•	NFR-006 시스템은 모든 핵심 데이터(세션 메타/승인/툴 결과/감사 로그)에 대해 백업 및 복구 전략을 제공해야 한다.
	•	NFR-007 시스템은 운영자가 툴/모델/정책을 배포 없이 관리할 수 있도록 구성(설정) 기반으로 제공해야 한다.
