이 문서는 특정 벤더/제품이 아니라, **엔터프라이즈 데이터 앱**에서 반복적으로 검증된 UI/UX 원칙을 정리한 레퍼런스 노트입니다. 구현은 `docs/frontend.md` 및 실제 프론트 코드/컴포넌트 기준으로 판단합니다.

## 디자인 시스템의 목표

- **일관성**: 여러 화면/워크플로우에서 동일한 상호작용 규칙과 시각 언어를 유지
- **확장성**: 기능/도메인이 늘어도 UI 컴포넌트가 기하급수적으로 늘지 않게 설계
- **재사용성**: 작은 프리미티브 조합으로 복합 UI를 만들 수 있게 구성
- **접근성**: WCAG 기반 대비/키보드 내비게이션/스크린리더 호환을 기본으로 고려

## 핵심 원칙

### 1) Composition / Reusability

- “드롭다운/검색 선택기/필터 빌더” 같은 복합 컴포넌트도 내부는 **Button + Popover + Menu + Input** 같은 프리미티브 조합으로 만들 수 있게 설계합니다.
- 결과적으로 컴포넌트 수가 폭증하지 않고, 패턴이 재사용되며, 사용자가 학습한 UX가 그대로 전파됩니다.

### 2) Accessibility (P0)

- 색상은 단순 RGB가 아니라 **의미 토큰**(primary/success/warning/danger 등)으로 사용합니다.
- 키보드 포커스/탭 순서/ESC 닫기/단축키 등은 “옵션”이 아니라 기본 UX로 포함합니다.

## 색상/테마

- 색상은 값이 아니라 **의미(intent) 토큰**으로 관리합니다.
- 라이트/다크 모드는 “스킨 덮어쓰기”가 아니라, 컴포넌트가 기본적으로 양쪽 테마를 지원하도록 구성합니다.
- 경고/실패/차단/DEGRADED 상태는 시각적으로 강하게(하지만 과도하게) 드러나야 합니다.

## 타이포그래피/레이아웃

- 전역 CSS 영향은 최소화하고, 텍스트 스타일은 클래스/컴포넌트로 옵트인합니다.
- 데이터 앱은 “가독성”과 “밀도(density)”가 공존해야 하므로, **Compact/Default/Comfortable** 같은 density 모드를 고려합니다.

## 데이터 앱 UI 패턴 (추천)

- **3-pane 레이아웃**: Left Nav / Main / Right Inspector
- **Command/Task 추적 Drawer**: 비동기 작업(202) 진행상태를 사용자가 스스로 추적 가능
- **표/그리드**: 정렬/필터/그룹/가상 스크롤/셀 편집 + 스키마 기반 검증
- **폼/검증**: “제출 전”에 에러를 잡는 즉시 피드백(스키마/권한/조건)
- **Diff/Preview**: “적용 전 미리보기”를 기본 UX로 제공(특히 writeback)
- **검색/바로가기**: Cmd/Ctrl+K 커맨드 팔레트, 전역 검색, 리소스 점프

## 엔지니어링 관점의 시사점

- UI 컴포넌트는 “화면 단위”가 아니라 “프리미티브 + 패턴” 단위로 설계하는 것이 유지보수에 유리합니다.
- 접근성/테마/단축키는 초기에 넣지 않으면 나중에 비용이 급증합니다.
- 대규모 데이터 UI 병목은 대부분 렌더링/DOM/테이블 가상화에서 발생하므로, 기본 구조를 먼저 설계합니다.
